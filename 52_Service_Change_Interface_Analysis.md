# 52. 서비스변동 (마이페이지/고객요청) 타시스템 연동 통합 분석서

## 1. 개요
본 문서는 `msf-api` 프로젝트 내 **서비스/정보 변동(부가서비스, 명의변경, 해지, 요금제 변경, 일시/분실정지 등)** 관련 전반적인 기능의 타시스템(M플랫폼, MSP, 자체 DB 및 공통 모듈) 연동 규격과 비즈니스 로직을 통합 분석하여 정의합니다.
기준 소스는 `form.servicechange` 패키지(AS-IS 및 TO-BE 변환 진행분)입니다.

## 2. 업무 유형별 연동 방식 분류

서비스 변동 업무는 **온라인 실시간 연동 (KOS 즉시 반영)** 업무와 **오프라인 접수 (수기 처리 대기)** 업무로 크게 나뉩니다.

1. **실시간 KOS 연동 대상** (M플랫폼 API 호출로 이통사 기간망(KOS)에 즉각 반영)
   - 부가서비스 가입/해지/변경
   - 요금제 변경 / 예약 / 취소
   - 일시정지 및 분실정지 / 해제
   - 유심 PUK 번호 조회 등

2. **오프라인 비동기 접수 대상** (MSF 자체 DB 적재 후 상담원(BOS) 수기 처리 대기)
   - 명의변경 (Name Change)
   - 서비스 일반 회선 해지 (Cancellation)
   - 요금내역 / 통화내역 열람 신청
   - 가입증명서 출력 대행 신청

## 3. 세부 업무별 연동 방향 및 인터페이스 분석

### 3.1. 요금제 변경 (MsfFarPricePlanController)
고객의 통신 요금제를 즉시 변경하거나 다음 달 1일 자로 변경을 예약하는 기능입니다.

*   **즉시 요금제 변경**
    *   **사용 전문**: `mPlatFormService.farPricePlanChg` (또는 `farPricePlanChgNeTrace`)
    *   **업무 목적**: 당일 즉시 이용 가능한 요금제로 변경하도록 KOS에 반영.
*   **예약 요금제 변경 및 예약 취소**
    *   **사용 전문**: `mPlatFormService.doFarPricePlanRsrvChg`, `doFarPricePlanRsrvCancel`
    *   **업무 목적**: 익월 결제 주기(1일)에 반영될 요금제 예약 접수 및 해당 예약 건의 취소 처리.
    *   **유의사항**: 적용 전 결제일, 위약금 발생 가능성, 기존 요금제 조건 조회 전문과 브리지(Bridge) 형태로 조합되어 사전 점검 로직을 거침.

### 3.2. 일시정지 / 분실정지 및 해제 (MsfPauseController)
스마트폰 분실 시 선조치, 또는 군입대/해외체류 등 장기 미사용 시 회선을 정지시키거나 복구하는 기능입니다.

*   **일반 일시정지**
    *   **조회/가능여부**: `suspenPosHis`(정지 이력조회), `suspenPosInfo`(정지 가능여부)
    *   **적용(정지)**: `suspenChg` (일반 일시정지 전문)
    *   **해제**: `suspenCnlPosInfo` (해제 가능조회), `suspenCnlChgIn` (해제 전문 반영)
*   **분실 정지**
    *   **조회**: `pcsLostInfo`
    *   **적용(분실)**: `pcsLostChg`
    *   **해제**: `pcsLostCnlChg`
    *   **특징**: 기기 분실이라는 보안성을 띠고 있어, 관련 고객 안심 서비스와 엮여 분실 정지 로직이 일반 정지와 완전히 분리되어 연동 됨.

### 3.3. 명의변경 / 서비스일반해지 / 증명서 신청 (MsfCustRequestController)
MVNO/이통사 정책상 신분증 사본 및 본인 확인, 위약금 정산 등 복합적인 확인이 필요한 보안 업무입니다. 이에 따라 앱/웹(온라인)에서 실시간으로 곧바로 완료(KOS Call)시킬 수 없고 **오프라인 접수 후 상담원 수기 검토(BOS)** 를 거칩니다.

*   **연동 방향**: M플랫폼 기간망 직접 연동 없음. 자체 DB(`MSF_CUST_REQUEST...` 시스템) 데이터 적재.
*   **처리 흐름 세부**:
    1. **요청 접수분류**: `reqType` 파라미터로 식별 (`NC`: 명의변경, `CL`: 통화내역, `JF`: 가입신청서 등)
    2. **내부 DB (접수함 적재)**: `insertCustRequestMst` 및 유형별 하위 테이블 Insert 처리 (상담원 인트라넷 인입용).
    3. **서식지 연동 (SCAN/ECM 서버)**: `custRequestScanService.prodSendScan()`
        - 고객이 전자 서명한 신청 서류나 신분증 사본 데이터를 이미지/PDF 형태의 메타 규격으로 포장하여, 중앙 문서보관용 SCAN 서버에 전송. 이후 상담원이 해당 서류를 육안 검토 후 전산 권한으로 KOS 오프라인 처리 진행.

### 3.4. 휴대폰 안심보험 예외 플로우 (MsfCustRequestController 내 `IS` 타입 처리)
일반 오프라인 고객 접수(Customer Request)로 들어오나, 실시간 KOS 승인이 혼재된 하이브리드 플로우를 지닙니다.
1. 고객 요청 DB 적재 및 SCAN 서류 전송 (보안/증빙 기록 적재 보관용).
2. KOS 부가서비스 실시간 다건 신청 전문을 태워 당일 즉각적용 처리.
3. 알림 서버(`SmsSvc`) 연동 파이프를 통해, 가입 자급제 단말 등에 연계된 안내 LMS(결과 문자)를 비동기식으로 발송 완료.

### 3.5. 부가서비스 (MsfRegSvcController) [참고 요약]
이통사 일반 부가서비스 및 로밍서비스 등의 실시간 가입/해지 (상세 별도 문서).
*   **주요 KOS 연동 전문**: `X97`(이용/조건 상세조회), `X38`(단건해지 및 개별이력해지), `Y25`(부가서비스 변경 및 복수적용 다건신청).
*   **주요 MSP 정책 검증**: `MSP_RATE_MST@DL_MSP` 단락조회를 통해, 요금제 종속형이나 포인트결제형(`REG_SVC_CD_4`) 상품의 온라인 강제 해지 가능여부(`onlineCanYn`) 사전 차단 로직 적용.

## 4. 공통 비기능 및 보안 검증사항 연동 (TOBE 분리 과제)

마이페이지 내 **회선 제어/변경** 행위는 통신사 핵심 보안이 요구되므로 AS-IS 서비스 레이어에 다양한 검증 기술이 결합되어 있습니다. TO-BE (MSA 전환) 시 API Gateway 혹은 전역 필터 계층(Filter Layer)으로 분리 설계되어야 합니다.

*   **본인인증 STEP 단계 상향 검증**: `certService.getStepCnt()` 및 `vdlCertInfo()`
    *   **접근 제어**: 일반 변경/정지/명의변경 요청 시에는 최소 NICE/PASS 본인인증 4단계(STEP 4) 세션 권한이 확인되어야 하며, 특히 프라이버시가 강한 "통화내역 열람(`CL`)" 업무는 최고 인증 단계인 **5단계(STEP 5)** 가 충족되어야 전문이 허가됩니다.
    *   **데이터 Check**: KOS 요청 전, 본인 여부 점검을 위해 세션에 남은 CI(연계정보), 계약번호, 그리고 암호화된 주민등록번호 등을 Controller 내에서 Cross Check 합니다.
*   **IP 및 상태 감사(Audit) 추적**: 타인의 번호 도용/변경 조작을 감시하기 위해, 호출자의 Client IP 로그 및 `McpIpStatisticDto` 객체를 통해 API의 주요 Request를 Admin Access Trace(보안 이력 테이블)로 상시 적재(Audit Logging)합니다.
