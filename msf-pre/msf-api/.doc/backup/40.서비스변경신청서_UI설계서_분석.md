## 2. 전체 플로우 및 동작 순서

### 2.1 플로우 요약

**MMSP-DS-02 v1.0 (20260310) 반영**  
- 가입유형 선택(S102010101)을 **고객(S102020101) 화면과 통합·하단 배치**. (서비스해지 20260310과 동일 패턴.)  
- **서비스 선택을 카테고리화**: 서비스 변경 옵션을 카테고리(대분류)별로 그룹화하여 표시. 사용자가 카테고리 단위로 인지하고 선택할 수 있도록 구성.

```
고객(S102020101 + 가입유형 선택 하단) → 상품(S102030101) → 동의(S102040101) → 신청완료(S102050101)
```

- **기존 4단계 플로우**(설계서 v0.6): 가입유형 선택(S102010101) → 고객(S102020101) → 상품(S102030101) → 동의(S102040101) → 신청완료(S102050101).
- **선택한 서비스 변경만** 신청서 작성(상품 단계에서 해당 영역만 노출).
- **동시 변경 불가 업무**: 번호변경, 분실복구/일시정지해제 신청, 요금상품 변경, 핸드폰 기변/일반 기변 → 이 중 **2개 이상 선택 불가**.
- **mform-web 현재 구현**: 1단계 `ChangeTypeCust.vue`에서 고객 정보 + 서비스 변경 유형(가입유형) 선택을 한 화면으로 처리. 진입 URL `/mobile/change/ChangeTypeCust`, 단계는 고객 → 상품 → 동의 3단계.

### 서비스 변경 대상(서비스 연결) 조회 연동 흐름 — 전체 확인

| 단계 | 위치 | 내용 |
|------|------|------|
| **1. 화면 진입** | `ChangeTypeCust.vue` | `onMounted` 시 `getChangeTargets()` 호출. |
| **2. API 호출** | `api/serviceChange.js` | `getChangeTargets()` → `msfGet('/v1/service-change/base/targets')` → 실제 요청: `GET /api/v1/service-change/base/targets`. |
| **3. 컨트롤러** | `SvcChgBaseController.java` | `@GetMapping("/targets")` → `svcChgBaseSvc.getChangeTargets()` 호출. |
| **4. 서비스** | `SvcChgBaseSvcImpl.java` | ① **우선**: `selectSvcChgTargetFromCodeGroup()` + `selectSvcChgTargetFromCodeDetails()` (MSF_CD_GROUP_BAS, MSF_CD_DTL, SVC_TGT_CD). ② 옵션 중 `EXPNSN_STR_VAL1='N'` → Mapper에서 `concurrentGroupId='CORE'`로 매핑. ③ `categoryList` = 그룹 1건 + 옵션 목록, `concurrentGroups` = CORE인 옵션 코드 리스트. ④ 실패 시 msf_svc_chg_ctg/opt 또는 상수 fallback. |
| **5. Mapper** | `SvcChgBaseMapper.xml` | `selectSvcChgTargetFromCodeGroup`: MSF_CD_GROUP_BAS에서 CD_GROUP_ID='SVC_TGT_CD' 1건 → id, label. `selectSvcChgTargetFromCodeDetails`: MSF_CD_DTL에서 SVC_TGT_CD 상세, EXPNSN_STR_VAL1='N'이면 concurrentGroupId='CORE'. |
| **6. DB** | 공통코드 | MSF_CD_GROUP_BAS (SVC_TGT_CD), MSF_CD_DTL (DTL_CD, DTL_CD_NM, **EXPNSN_STR_VAL1** = 동시변경가능여부 Y/N). INSERT: `tabdoc/MSF_CD_GROUP_BAS_CD_DTL_insert.sql`. |
| **7. 화면 반영** | `ChangeTypeCust.vue` | `res.categoryList` → `categoryList.value` (카테고리 라벨 + 옵션 그리드). `res.concurrentGroups` → `coreChangeCodes.value = res.concurrentGroups.flat()` (동시 선택 불가 표시·검증용). |
| **8. 검증** | 동일 컴포넌트 | `hasInvalidCombination`, `isOptDisabled`, `toggleSelectAll` 등에서 `coreChangeCodes` 사용. 서비스 체크 시 백엔드 `validateServiceCheck`에서도 동시 불가 코드는 `getConcurrentDisallowedCodes()`로 MSF_CD_DTL(SVC_TGT_CD) 재조회 후 EXPNSN_STR_VAL1='N' 기준 검증. |

**정리**: 화면에서 부르는 **서비스 연결(대상 목록)** 은 `GET /api/v1/service-change/base/targets` 한 번으로, **MSF_CD_GROUP_BAS·MSF_CD_DTL (SVC_TGT_CD)** 조회 결과를 받아 카테고리·옵션 표시와 **동시변경 가능 여부(EXPNSN_STR_VAL1)** 를 셋팅해 사용한다.

| **상태 분리** | (1) `form` 고객 정보 (2) `selectedIds` 서비스 옵션 (3) `phoneAuthCompleted`, `serviceCheckCompleted` — 서비스 체크 완료 후 옵션 변경 시 `serviceCheckCompleted` 초기화 여부(설계: 선택 변경 시 재확인 유도) |
| **고객유형 → 필드 세트** | `custType` 값에 따라 노출 필드 전환: NA/FN → 이름+생년월일+성별; NM/FNM → 이름+생년월일(또는 외국인등록번호)+법정대리인; JP/PP → 이름+법인등록번호+사업자등록번호+방문고객. **방문고객**은 JP/PP일 때만 노출, NA/FN은 본인 고정, NM/FNM은 대리인 고정 |
| **인증 후 잠금** | 인증 성공 시 변경 휴대폰번호·이름·생년월일(또는 법인등록번호) `readonly` 또는 `disabled`, [인증] → [인증 완료] 문구·비활성. **연락처 휴대폰**은 인증 완료 번호와 동일값이면 readonly로 유지할지 설계 확인 |
| **동시 변경 불가** | `coreChangeCodes` 4개 중 2개 이상 선택 시 즉시 에러 문구 표시 + [다음] 비활성. **전체선택** 시 이 4개는 제외하고 나머지만 체크. 옵션 개별 해제 시 에러 사라지는지 |
| **서비스 체크 버튼** | 1개 이상 선택 + 동시 변경 불가 미위반일 때만 [서비스 체크] 활성. 클릭 시 `serviceCheckCompleted = true`. [다음]은 인증+필수입력+**서비스 체크 완료**까지 모두 만족 시에만 활성 |
| **대리점** | select 옵션 목록 출처(API/공통코드). 필수 아님. 1단계에서 미선택 시 상품/동의 단계에서 필수인지 설계서 추가 확인 |
