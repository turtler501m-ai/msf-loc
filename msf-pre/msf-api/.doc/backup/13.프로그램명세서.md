## 2. 프로세스 처리 구조 (전체)

### 2.1 처리 흐름 개요

```
[사용자] → [라우터] → [McpMainLayout] → [McpServiceDetailView]
                              │
              ┌───────────────┴───────────────┐
              │ route.path = /mobile/:domain/:service
              │ (예: /mobile/change/ChangeType)
              ▼
    ┌─────────────────────────────────────────┐
    │ 메뉴 스토어로 현재 메뉴가 "스텝형" 인지 판별   │
    │ (stepSubMenu && children 존재)           │
    └─────────────────────────────────────────┘
              │
    ┌─────────┴─────────┐
    │ YES: 스텝형 메뉴   │  NO: 단일 컴포넌트
    │ (신규/변경,       │     (부가기능 등)
    │  서비스변경,      │
    │  명의변경,        │
    │  서비스해지)      │
    ▼                   ▼
[McpStepService]    [getRoutingComponent(route)]
    │                   → components/{domain}/{service}.vue
    │
    ├─ changeAllSteps(route.path)
    │     → mcp_menu에서 해당 메뉴 children 조회
    │     → children[] → step 컴포넌트 배열 생성 (getStepComponent)
    │     → allSteps = [ ChangeType, ChangeCust, ChangeProd, ChangeAgree ]
    │
    ├─ currentStepIndex = 0..N (진행 단계)
    ├─ visibleSteps = allSteps.slice(0, currentStepIndex + 1)  // 밑으로 쌓이는 UI
    │
    ├─ [다음] 클릭 → currentStepIndex++ (검증은 임시 생략)
    └─ [작성완료] 클릭 (마지막 스텝) → stepRefs[last].save() 호출
              │
              ▼
    [*Agree.vue save()]
    → confirm → setLastCompletedName(name) → store.reset() → router.push(complete)
              │
              ▼
    [ServiceCompleteView] (route: /mobile/complete/:domain)
    → domain별 store에서 lastCompletedCustomerName 조회·마스킹 표시
    → 신청서 열람/톡/SMS/다운로드 안내 (연동 예정)
    → [홈으로] → clearLastCompletedName() → router to /
```

### 2.2 라우팅·메뉴·컴포넌트 매핑

| 경로 패턴 | 컴포넌트 | 비고 |
|-----------|----------|------|
| `/` | McpHomeView | 메인 |
| `/mobile/:domain/:service` | McpServiceDetailView | domain=change|ident|cancel|regist|addition, service=컴포넌트명 |
| `/mobile/complete/:domain` | ServiceCompleteView | domain=change|ident|cancel |

**스텝형 메뉴 (mcp_menu.js)**  
- `id: '02'` 서비스변경 → url `/mobile/change/ChangeType`, children step 0~3  
- `id: '03'` 명의변경 → url `/mobile/ident/IdentType`, children step 0~3  
- `id: '04'` 서비스해지 → url `/mobile/cancel/CancelType`, children step 0~3  

**동적 컴포넌트 로딩 (mcp_components.js)**  
- `getStepComponent(subMenuUrl)`: URL에서 `domain/service` 추출 → `@/components/{domain}/{service}.vue`  
- `getRoutingComponent(route)`: `route.params.domain`, `route.params.service` → 동일 경로 규칙  

### 2.3 스텝별 처리 단위 (공통 패턴)

| 스텝 인덱스 | 스텝명 | 역할 | Store 연동 | API 호출 (예) |
|-------------|--------|------|------------|---------------|
| 0 | 가입유형 선택 | 대리점/변경항목 선택 | setAgencyCode / setSelectedOptions | (선택 시) 동시처리불가 체크 등 |
| 1 | 고객 | 고객정보·휴대폰 인증 | setCustomerForm | getJoinInfo (X01) |
| 2 | 상품 | 서비스/상품·부가·납부 등 | setProductForm | getCurrentAddition(X20), getRemainCharge 등 |
| 3 | 동의 | 안내·서명·확인 | agreeForm | (작성완료 시) 신청서 등록·이미징·전송 |

각 스텝 컴포넌트는  
- `emit('update:complete', isComplete)` 로 유효성 상태 전달  
- `defineExpose({ validate, save })` 로 레이아웃에서 검증/저장 호출  
- `onMounted`에서 해당 Store에서 복원 후 `watch`로 Store 동기화  

### 2.4 X01 가입정보조회 (join-info) 상세

**목적**: M포털 X01 가입정보조회와 동일 스펙. 휴대폰 번호(ctn) + 이름으로 본인 확인 후 계약번호(ncn)·고객번호(custId) 확보, 이후 Y04 계약정보 유효성체크로 인증 완료.

| 구분 | 내용 |
|------|------|
| **API 코드** | X01 (가입정보조회), Y04 (계약정보 유효성체크) |
| **엔드포인트** | 서비스변경: `POST /api/v1/join-info` · 명의변경: `POST /api/v1/ident/join-info` · 서비스해지: `POST /api/v1/cancel/join-info` · 부가서비스변경: `POST /api/v1/addition/join-info` |
| **요청 필수** | `name`(고객명), `ctn`(휴대폰번호 10~11자리) |
| **응답** | `success`, `ncn`, `custId`, `addr`, `email`, `homeTel`, `initActivationDate`, 실패 시 `message` |

**처리 흐름 (백엔드)**  
1. **입력 검증**: ctn 필수·형식(숫자 10~11자리), name 필수  
2. **M전산 조회**: `SvcChgMypageMapper.selectContractInfo(name, mobileNo)` → `msp_juo_sub_info` (또는 운영 시 `MSP_JUO_SUB_INFO@DL_MSP`)에서 `contract_num`(ncn), `customer_id`(custId) 조회. 조건: `sub_status <> 'C'`, 휴대폰·이름 일치  
3. **Mock (로컬)**: `SERVER_NAME=LOCAL` 이고 `mform.join-info.mock-when-no-db=true` 이면 DB 실패 시 ncn/custId Mock 세팅  
4. **Y04 호출**: `MplatFormSvc.contractValdChk(clientIp, cIntUsrId, ncn, ctn, custId)` → M플랫폼 계약정보 유효성체크. LOCAL이면 Mock 성공  
5. **응답 세팅**: success=true 시 ncn, custId, addr, email, homeTel, initActivationDate 반환 (addr/email/initActivationDate는 현재 샘플 고정값)

**관련 소스**  
| 레이어 | 파일 | 역할 |
|--------|------|------|
| Controller (연동 공통) | `formComm/controller/SvcChgRestController.java` | join-info, ident/join-info, cancel/join-info, addition/join-info, service-change/concurrent-check |
| Controller | `formOwnChg/controller/OwnChgController.java` | ident/eligible, ident/addition/current, ident/apply, ident/grantor-req-chk, ident/check-tel-no |
| Controller | `formSvcCncl/controller/CancelController.java` | cancel/eligible, cancel/remain-charge, cancel/apply |
| Controller | `formSvcChg/controller/SvcChgRegController.java` | addition/current, addition/reg, addition/cancel |
| Service (공통) | `formComm/service/JoinInfoSvcImpl.java` | `joinInfo()`: 검증·M전산·Y04·VO 세팅 (서비스변경·명의변경·해지·부가서비스변경 공통) |
| Service | `formSvcChg/service/SvcChgInfoSvcImpl.java` | `joinInfo()` → formComm.JoinInfoSvc 위임 |
| Service | `formOwnChg/service/OwnChgInfoSvcImpl.java` | `joinInfo()` → SvcChgInfoSvc.joinInfo() 위임 |
| Mapper | `formComm/mapper` → `ContractInfoMapper.xml` | `selectContractInfo` → msp_juo_sub_info 조회 |
| M플랫폼 | `common/mplatform/MplatFormSvc.java` | `contractValdChk()` Y04 호출 (LOCAL 시 Mock) |
| DTO/VO | `formComm/dto`: JoinInfoReqDto, JoinInfoResVO, SvcChgInfoDto(고객컨텍스트) | 연동·업무 공통; join-info 응답은 JoinInfoResVO |

**프론트 호출**  
- 서비스변경: `ChangeTypeCust.vue` → `getJoinInfo()` (serviceChange.js 또는 동일 API) → 성공 시 `formStore.setCustomerForm({ ncn, custId, phoneAuthCompleted: true, ... })`

- 부가서비스변경: `/api/v1/addition/join-info` 호출 시 동일 formComm.JoinInfoSvc(Y04) 사용

**ASIS 대비**  
- M포털: `MplatFormService.perMyktfInfo()` (eventCd X01), `MypageMapper.selectContractNum` (이름+휴대폰 → CONTRACT_NUM).  
- TO-BE: X01 “가입정보조회”는 이름+ctn → M전산(ncn, custId) → Y04 까지 한 번에 수행하는 **join-info** 한 API로 제공되며, 응답에 ncn/custId를 담아 상품 단계(X20, X97 등)에서 재사용.

## 3. 서비스별 프로세스 상세

### 3.1 서비스변경 신청서 (domain=change)

```
[ChangeType] step 0
  ├─ 선택: selectedOptions (복수) → setSelectedOptions()
  ├─ (선택 시) checkConcurrentChange(selectedCodes) 호출 가능
  └─ 완료 조건: selectedOptions.length > 0

[ChangeCust] step 1
  ├─ 폼: 고객유형, 이름, 생년월일, 휴대폰, 이메일, 주소, 법정대리인/대리인 정보 등
  ├─ [인증] 클릭 → getJoinInfo({ ctn }) → 성공 시 setCustomerForm({ phoneAuthCompleted: true, ... })
  ├─ watch(form, → setCustomerForm)
  └─ onMounted: store.customerForm 복원

[ChangeProd] step 2
  ├─ 폼: 부가서비스, 요금변경, 번호변경, 분실복구, 단말보험, 기변, USIM, 데이터쉐어링, 아무나SOLO 등
  ├─ 부가/팝업: McpTmBlockPop, McpNumberTheftBlockPop, McpRoamingAllDayPop, McpAdditionEditPop 등
  ├─ watch(productForm, → setProductForm)
  └─ onMounted: store.productForm 복원

[ChangeAgree] step 3
  ├─ 신청정보 요약: formStore.selectedOptions, customerForm, productForm 표시
  ├─ 서명: signAgreed (eFormSign 연동 예정)
  ├─ validate(): 서명 완료 여부
  └─ save(): confirm → setLastCompletedName(name) → reset() → push(complete/change)
```

**Store (service_change_form.js)**  
- state: selectedOptions, customerForm, productForm, agreeForm, lastCompletedCustomerName  
- reset() 시 lastCompletedCustomerName 제외 초기화  

### 3.2 명의변경 신청서 (domain=ident)

```
[IdentType] step 0
  ├─ 대리점 선택: agencyCode → setAgencyCode()
  └─ 완료 조건: agencyCode 존재

[IdentCust] step 1
  ├─ 양도/양수 고객, 방문유형, 대리인 정보 등
  ├─ [인증] → getJoinInfo (ident API) → setCustomerForm
  └─ watch / onMounted: ident_form Store 동기화

[IdentProd] step 2
  ├─ 유심, 요금제, 할인, 부가서비스, 납부정보
  ├─ onMounted: getCurrentAddition(ctn) → additionItems 반영, setProductForm
  └─ watch → setProductForm

[IdentAgree] step 3
  └─ save(): setLastCompletedName → reset() → push(complete/ident)
```

**Store (ident_form.js)**  
- state: agencyCode, customerForm, productForm, agreeForm, lastCompletedCustomerName  
- productForm.additionItems 는 API 조회 결과 반영  

### 3.3 서비스해지 신청서 (domain=cancel)

```
[CancelType] step 0
  └─ 대리점 선택 → setAgencyCode()

[CancelCust] step 1
  └─ getJoinInfo (cancel API) → setCustomerForm

[CancelProd] step 2
  ├─ onMounted: getRemainCharge(ctn) → remainCharge, penalty, installmentRemain 반영
  └─ watch → setProductForm

[CancelAgree] step 3
  └─ save(): setLastCompletedName → reset() → push(complete/cancel)
```

**Store (cancel_form.js)**  
- state: agencyCode, customerForm, productForm(remainCharge, penalty, installmentRemain 등), agreeForm, lastCompletedCustomerName

## 4. 작성완료 ~ 신청완료 화면 프로세스

### 4.1 시퀀스

```
[사용자] [작성완료] 클릭
    │
    ▼
McpStepService.clickCompelteBtn()
    │ → stepRefs[currentStepIndex].save() 호출 (마지막 스텝 = *Agree)
    ▼
*Agree.save()
    │ 1. confirm('신청서를 등록하시겠습니까?') → 취소 시 return
    │ 2. (실연동 시) 신청서 등록 API, 이미징, M플랫폼 전송, SMS/카톡
    │ 3. formStore.setLastCompletedName(customerForm.name)
    │ 4. formStore.reset()  // lastCompletedCustomerName은 reset에서 초기화하지 않음
    │ 5. router.push({ name: 'service-complete', params: { domain } })
    │ 6. 실패 시 alert('신청서 등록이 실패하였습니다...')
    ▼
ServiceCompleteView (route.params.domain = change|ident|cancel)
    │ 1. domain별 Store에서 lastCompletedCustomerName 조회
    │ 2. maskName(name) → "홍길*" 형태 메시지 표시
    │ 3. 신청서 열람 / 톡 발송 / SMS 발송 / 다운로드 안내 (UI만, 연동 예정)
    ▼
[홈으로] 클릭
    │ → domain별 store.clearLastCompletedName()
    │ → router-link to="/"
    ▼
메인 화면
```

### 4.2 lastCompletedCustomerName 처리 규칙

- **설정**: `*Agree.save()` 내에서 `setLastCompletedName(customerForm?.name)` 호출  
- **유지**: `reset()` 에서는 lastCompletedCustomerName 을 초기화하지 않음 (완료 화면에서 사용)  
- **삭제**: `ServiceCompleteView` 에서 [홈으로] 클릭 시 `clearLastCompletedName()` 호출

## 5. API 연동 구조

### 5.1 프론트 → 백엔드

| 계층 | 파일 | 역할 |
|------|------|------|
| 공통 | api/mform.js | BASE_URL(/api), mformFetch, mformPost, mformGet |
| 서비스변경 | api/serviceChange.js | getJoinInfo, checkConcurrentChange, getCurrentAddition, regAddition, cancelAddition |
| 명의변경 | api/ident.js | getJoinInfo, getCurrentAddition, checkIdentEligible |
| 서비스해지 | api/cancel.js | getJoinInfo, getRemainCharge, checkCancelEligible |

**호출 시점**  
- **getJoinInfo**: 고객 스텝에서 [인증] 클릭 시 (ChangeCust, IdentCust, CancelCust)  
- **getCurrentAddition**: ChangeProd(필요 시), IdentProd onMounted  
- **getRemainCharge**: CancelProd onMounted  
- **checkConcurrentChange**: ChangeType 선택 변경 시 (구현 시)  
- **checkIdentEligible / checkCancelEligible**: 인증 후 또는 다음 스텝 진입 시 (구현 시)  

### 5.2 백엔드 API (mform-api)

| 메서드 | 경로 | 용도 | 비고 |
|--------|------|------|------|
| POST | /api/v1/join-info | X01 가입정보조회 (Y04) | 서비스변경, formComm 공통 |

| POST | /api/v1/cancel/join-info | X01 (서비스해지) | joinInfo 재사용 |
| POST | /api/v1/addition/join-info | 부가서비스변경 휴대폰 인증 (Y04) | formComm.JoinInfoSvc 공통 |
| POST | /api/v1/service-change/concurrent-check | 동시 변경 불가 체크 | Mock |
| POST | /api/v1/addition/current | X20 부가서비스 조회 | Mock |
| POST | /api/v1/ident/addition/current | 명의변경 부가 조회 | additionCurrent 재사용 |
| POST | /api/v1/ident/eligible | 명의변경 가능 여부 | Mock |
| POST | /api/v1/cancel/eligible | 서비스해지 가능 여부 | Mock |
| POST | /api/v1/cancel/remain-charge | 잔여요금/위약금/분납금 | Mock, X22/X67 연동 예정 |

프론트 baseURL이 `/api` 이므로 Vite proxy로 `http://localhost:8081` 등에 전달되는 구조 가정.

## 7. 레이아웃·공통 컴포넌트 처리 구조

### 7.1 McpStepService.vue

| 항목 | 내용 |
|------|------|
| 입력 | route.path (현재 URL) |
| 처리 | getCurrentParentMenu(path) → children으로 allSteps 생성, getStepComponent(sub.url)로 컴포넌트 생성 |
| 상태 | currentStepIndex, isCurrentStepComp[], stepRefs[] |
| UI | visibleSteps 만 렌더 (0~currentStepIndex 까지 세로 누적 표시) |
| 버튼 | [다음]: currentStepIndex 증가 (검증 임시 생략) / [작성완료]: 마지막 스텝 ref.save() |
| expose | 없음 (내부에서 stepRefs[i].validate(), stepRefs[i].save() 호출) |

자식 스텝 컴포넌트 규약:  
- `@update:complete` 로 유효성 전달  
- `validate()`: Promise<{ valid, message }>  
- `save()`: 작성완료 시 호출 (마지막 스텝만)  

### 7.2 McpServiceDetailView.vue

- route.path로 메뉴 스토어 조회  
- stepSubMenu && children 있으면 `<mcp-step-service>` 렌더  
- 없으면 `getRoutingComponent(route)` 로 단일 컴포넌트 렌더  

### 7.3 팝업·공통

## 9. 에러·예외 처리
