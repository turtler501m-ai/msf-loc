# AS-IS 휴대폰 인증·가입정보조회 분석

스마트서식지 TO-BE 개발 참고를 위해 MCP(AS-IS)의 **휴대폰 인증** 및 **가입정보조회** 처리 흐름을 정리한 문서이다.

---

## 1. 개요

| 구분 | 설명 |
|------|------|
| **휴대폰 인증** | 고객이 입력한 **이름 + 휴대폰번호**로 M전산에서 계약정보를 조회한 뒤, SMS 인증 또는 NICE 인증으로 본인 확인을 수행하는 흐름 |
| **가입정보조회** | 이미 확보한 **ncn(계약번호), ctn(휴대폰번호), custId(고객번호)**로 M플랫폼 **X01** API를 호출해 주소·이메일·가입일 등 가입정보를 조회하는 기능 |

- **ncn** = 계약번호 (CONTRACT_NUM, 서비스계약번호 9자리)
- **ctn** = 휴대폰번호 (SUBSCRIBER_NO, 11자리)
- **custId** = 고객ID (CUSTOMER_ID)  
→ ncn, custId는 고객이 직접 알 수 없고, **이름+휴대폰번호**로 M전산에서 조회해야 한다.

---

## 2. 휴대폰 인증 흐름 (이름 + 휴대폰번호 → 계약정보 조회)

### 2.1 계약번호 조회 (이름 + 휴대폰번호)

**역할**: 로그인 전 또는 회선 추가 시, “이름 + 휴대폰번호”로 M전산에서 **계약번호(contractNum)** 를 조회한다.

| 계층 | 소스 위치 | 설명 |
|------|-----------|------|
| **Controller** | `mcp-portal-was` `MypageUserController.java` | 이름·휴대폰번호 수신 후 `mypageUserService.selectContractNum(name, phone)` 호출 |
| **Service** | `MypageUserServiceImpl.java` | `apiInterfaceServer + "/mypage/contractNum"` 로 POST (params: name, mobileNo) |
| **API** | `mcp-api` `MypageController.java` | `@RequestMapping("/mypage/contractNum")` → `mypageMapper.selectContractNum(map)` |
| **Mapper** | `mcp-api` `MypageMapper.xml` | `selectContractNum` |

**M전산 조회 (MypageMapper.xml)**

```xml
<select id="selectContractNum" parameterType="java.util.Map" resultType="java.lang.String">
    SELECT CONTRACT_NUM, SUBSCRIBER_NO AS MOBILE_NO
    FROM MSP_JUO_SUB_INFO@DL_MSP
    WHERE SUB_STATUS <![CDATA[<>]]> 'C'
      AND SUBSCRIBER_NO = TRIM(#{mobileNo})
      AND UPPER(REPLACE(SUB_LINK_NAME,' ','')) = UPPER(REPLACE(#{name},' ',''))
</select>
```

- **MSP_JUO_SUB_INFO@DL_MSP**: M전산 가입자 정보
- **입력**: `name`(가입자명), `mobileNo`(휴대폰번호)
- **출력**: `CONTRACT_NUM` (계약번호) 1건
- **조건**: `SUB_STATUS <> 'C'` (해지 제외), 이름·휴대폰번호 일치

**사용 예 (MypageUserController)**

- 회선 추가 전 휴대폰 인증 요청 시: `selectContractNum(name, phone)` → 계약번호 없으면 "등록되어 있지 않은 경우" 등 처리
- 인증 통과 시: `certService.vdlCertInfo("C", certKey, certValue)` 로 인증 스텝 저장, 응답에 `contractNum` 포함

---

### 2.2 비로그인 회선정보 조회 (cntrListNoLogin)

**역할**: 로그인 없이 **휴대폰번호** 또는 **서비스계약번호**만으로 해당 회선의 계약·고객 정보를 한 건 조회할 때 사용한다.

| 계층 | 소스 위치 | 설명 |
|------|-----------|------|
| **호출처** | `mcp-portal-was` `MypageServiceImpl`, `PersonalServiceImpl`, `RegSvcServiceImpl` 등 | `RestTemplate.postForObject(apiInterfaceServer + "/mypage/cntrListNoLogin", userCntrMngDto, McpUserCntrMngDto.class)` |
| **API** | `mcp-api` `MypageController.java` | `@RequestMapping("/mypage/cntrListNoLogin")` → `mypageMapper.selectCntrListNoLogin(param)` |
| **Mapper** | `MypageMapper.xml` | `selectCntrListNoLogin` |

**입력 DTO**: `McpUserCntrMngDto`  
- `svcCntrNo`: 서비스계약번호 (ncn)  
- `cntrMobileNo`: 휴대폰번호  
- `subLinkName`: 가입자명 (선택)

**M전산 조회 (MypageMapper.xml)**

```xml
<select id="selectCntrListNoLogin" ... resultType="com.ktmmobile.mcp.mypage.dto.McpUserCntrMngDto">
    SELECT
        A.SVC_CNTR_NO
        , A.CONTRACT_NUM
        , A.CUSTOMER_ID AS CUST_ID
        , A.SUBSCRIBER_NO AS CNTR_MOBILE_NO
        , A.SUB_STATUS
        , A.SUB_LINK_NAME AS USER_NAME
        , B.CUSTOMER_ID
        , B.CUSTOMER_TYPE
        , ...
    FROM MSP_JUO_SUB_INFO@DL_MSP A, MSP_JUO_CUS_INFO@DL_MSP B
    WHERE A.CUSTOMER_ID = B.CUSTOMER_ID
      AND ( A.SVC_CNTR_NO = #{svcCntrNo} OR A.SUBSCRIBER_NO = TRIM(#{cntrMobileNo}) OR ... )
      AND SUB_STATUS <![CDATA[<>]]> 'C'
</select>
```

- **출력**: `SVC_CNTR_NO`, `CONTRACT_NUM`, `CUST_ID`, `CNTR_MOBILE_NO`, `USER_NAME`, `CUSTOMER_TYPE` 등 → 이후 M플랫폼 API 호출 시 **ncn, ctn, custId**로 사용

---

### 2.3 계약정보 상세 조회 (selectContractObj)

**역할**: 계약번호·이름·휴대폰·고객ID 등 조건으로 M전산에서 계약·청구·고객 유형 등을 Map으로 조회한다. NICE 등 본인인증 후 **계약정보와 인증정보 일치 여부** 검증할 때 사용한다.

| 계층 | 소스 위치 | 설명 |
|------|-----------|------|
| **Controller** | `EsimController.getContractInfoAuth` | `mypageUserService.selectContractObj("", "", contractNum)` 호출 |
| **Service** | `MypageUserServiceImpl.selectContractObj` | `apiInterfaceServer + "/mypage/selectContractObj"` POST (name, mobileNo, contractNum) |
| **API** | `mcp-api` `MypageController.java` | `@RequestMapping("/mypage/selectContractObj")` → `mypageMapper.selectContractObj(paraMap)` |
| **Mapper** | `MypageMapper.xml` | `selectContractObj` |

**M전산 조회 (MypageMapper.xml)**

- **테이블**: `MSP_JUO_SUB_INFO@DL_MSP`, `MSP_JUO_BAN_INFO@DL_MSP`, `MSP_JUO_CUS_INFO@DL_MSP`
- **출력**: `CONTRACT_NUM`, `SVC_CNTR_NO`, `SUBSCRIBER_NO`, `USER_SSN`, `SUB_LINK_NAME`, `CUSTOMER_ID`, `CUSTOMER_TYPE` 등
- **용도**: `getContractInfoAuth`에서 계약번호로 조회한 이름·주민번호와 NICE 인증 결과(이름·생년월일)를 비교해 일치 시 인증 성공 처리

---

### 2.4 휴대폰 인증 후 처리 요약

1. **이름 + 휴대폰번호** 입력
2. **selectContractNum(name, mobileNo)** → M전산에서 `CONTRACT_NUM` 조회  
   - 없으면: "사용자 정보가 일치하지 않습니다" 등 처리
3. (선택) **선납여부 등 추가 검증**  
   - 예: `mypageUserService.selectPrePayment(contractNum)`
4. **SMS 인증** 또는 **NICE 인증** 진행
5. 인증 성공 시 **certService.vdlCertInfo** 로 스텝 저장, 응답에 **contractNum** 포함
6. 이후 화면/API에서는 이 **contractNum**과 (필요 시 **cntrListNoLogin**으로 조회한 **ncn, ctn, custId**)를 사용해 M플랫폼 API 호출

## 3. 가입정보조회 (X01 – perMyktfInfo)

**역할**: 이미 알고 있는 **ncn, ctn, custId**로 M플랫폼 **X01**을 호출해, 해당 회선의 **주소·이메일·전화번호·가입일** 등 가입정보를 조회한다.

### 3.1 호출 경로

| 계층 | 소스 위치 | 설명 |
|------|-----------|------|
| **M플랫폼** | `mcp-portal-was` `MplatFormService.java` | `perMyktfInfo(ncn, ctn, custId)` → `getParamMap(ncn, ctn, custId, "X01")` → `mplatFormServerAdapter.callService(param, vo)` |
| **VO** | `MpPerMyktfInfoVO.java` | X01 응답 파싱: `addr`, `email`, `homeTel`, `initActivationDate` 등 |
| **사용처** | `MyinfoServiceImpl`, `PaywayServiceImpl`, `BillWayChgController`, `MyOllehController`, `PersonalBillController` 등 | `myinfoService.perMyktfInfo(ncn, ctn, custId)` 또는 `mPlatFormService.perMyktfInfo(...)` |

### 3.2 MplatFormService.perMyktfInfo

```java
public MpPerMyktfInfoVO perMyktfInfo(String ncn, String ctn, String custId) throws ... {
    MpPerMyktfInfoVO vo = new MpPerMyktfInfoVO();
    HashMap<String, String> param = getParamMap(ncn, ctn, custId, "X01");
    // LOCAL이면 getVo(1, vo), 아니면 mplatFormServerAdapter.callService(param, vo);
    return vo;
}
```

- **입력**: ncn(계약번호 9자리), ctn(전화번호 11자리), custId(고객ID)
- **출력**: `MpPerMyktfInfoVO` (addr, email, homeTel, initActivationDate 등)
- **비고**: 마스킹 해제 여부에 따라 `MpPerMyktfInfoVO.parse()`에서 주소·이메일·전화번호 마스킹 처리

### 3.3 사용 예

- **MyinfoController**: 청구지주소, 가입일 등 표시
- **PaywayController**: 결제수단 변경 시 주소 등
- **RegSvcController.myAddSvcListAjax** 등: 부가서비스 조회 전/후 가입정보 보강

## 4. Y04(계약정보 유효성체크)에 대한 AS-IS 코드 정리

기능명세서(15번 등)에는 다음이 명시되어 있다.

- **M플랫폼 API**: 계약정보 유효성체크 **Y04**
- **인터페이스**: IF_0179 / Y04
- **M플랫폼 서비스**: `MoscSvcContService.moscSvcContValdChk`
- **흐름**: M전산에서 이름+휴대폰번호로 custId, ncn 조회 → custId, ncn, 사용자 입력 ctn, 기타 필수값으로 **Y04** 호출

다만 **mcp-portal-was** Java 소스 내에서는 **Y04** 또는 **moscSvcContValdChk**를 호출하는 코드가 검색되지 않는다.  
실제 AS-IS에서는 다음으로 “계약정보 확인”을 수행하는 것으로 보인다.

- **M전산**: `selectContractNum` / `selectCntrListNoLogin` / `selectContractObj` 로 계약·고객 정보 조회
- **본인확인**: NICE 등 인증 후 `selectContractObj` 결과와 인증 정보(이름·생년월일) 비교  
  - 예: `EsimController.getContractInfoAuth`
- **가입정보**: 조회된 ncn, ctn, custId로 **X01(perMyktfInfo)** 호출

따라서 TO-BE에서 Y04를 그대로 연동할 경우, M플랫폼 규격서(연동규격서_MVNO-Selfcare-API_v4.84.xlsx 등)와 실제 M플랫폼 제공 여부를 확인한 뒤, 필요 시 **Y04**를 신규 연동하는 방식으로 설계하는 것이 좋다.

## 5. TO-BE 참고 사항 (스마트서식지)

- **join-info (계약정보 유효성·고객정보)**  
  - 18번 문서 등: `SvcChgRestController.joinInfo()` POST `/join-info`  
  - Y04 호출·고객유형·ncn·ctn·custId 반영  
  - M전산 연동: `SvcChgMypageMapper.selectContractInfo()` 등으로 계약번호·custId 조회 후 Y04/X01 연동 설계

- **동일하게 유지할 점**  
  - “이름 + 휴대폰번호” → M전산에서 계약번호(및 필요 시 custId) 조회  
  - 조회한 ncn, ctn, custId로 X01(가입정보) 호출  
  - 인증 스텝·본인확인 정책은 AS-IS와 동일하게 설계 가능

- **Y04**  
  - 규격서 기준으로 Y04를 “계약정보 유효성체크”로 도입할 경우, 위 2.1~2.3 단계 이후 또는 인증 완료 시점에 Y04 호출을 추가하는 형태로 정리할 수 있다.
