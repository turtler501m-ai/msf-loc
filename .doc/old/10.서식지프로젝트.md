### 3. 개발 적용 권장사항

1. **패키지 루트**  
   - 문서상 `msf` vs 현재 `mform`: 스마트서식지 전용 백엔드이므로 `com.ktmmobile.mform` 유지해도 무방.  
   - 규칙을 엄격히 적용하려면 `com.ktmmobile.msf` 로 통일하고, 기존 mform 패키지는 msf 하위로 리팩터링.

2. **도메인 패키지 구성 (mform-api)**  
   - 규칙의 하위 패키지를 도입할 때 예시:  
     - `com.ktmmobile.mform.formComm` — 신청서 공통, 솔루션 연동  
     - `com.ktmmobile.mform.formSvcChg` — 서비스변경 신청서  
     - `com.ktmmobile.mform.formOwnChg` — 명의변경 신청서  
     - `com.ktmmobile.mform.formSvcCncl` — 서비스해지 신청서  
   - 기존 `controller.v1`, `common`, `dto` 는 그대로 두고, **신규 기능**을 formSvcChg/formOwnChg/formSvcCncl 등으로 나누어 추가.

3. **파일명 규칙**  
   - **신규 작성 시**  
     - Controller: 연동 API → `XXXRestController.java`, 내부 전용 → `XXXController.java`  
     - Service: `서비스명칭Svc.java`, `서비스명칭SvcImpl.java`  
     - Dao: `서비스명칭Dao.java`, `서비스명칭DaoImpl.java`  
     - DTO: 업무별 `서비스명칭Dto.java`, `서비스명칭VO.java`, XML/Map 처리 시 `서비스명칭XmlWrapper.java` 등  
   - 기존 `MplatFormService` 등은 연동 공통이므로 유지하고, **신규 서비스**부터 Svc/SvcImpl 규칙 적용.

4. **메소드명**  
   - DB CRUD가 생기는 레이어(Dao/Service)에서는  
     `insert서비스명칭`, `save서비스명칭`, `update서비스명칭`, `delete서비스명칭`, `select서비스명칭`, `selectList서비스명칭` 형태로 통일.

5. **Mapper XML**

## 리팩터링 계획 (com.ktmmobile.msf 적용)

- **목표**: mform-api 를 문서 규칙(com.ktmmobile.msf, Controller/RestController, formComm 등)에 맞게 리팩터링. **M플랫폼 연동(MplatFormService, MplatFormServerAdapter) 로직은 그대로 유지**하고 패키지·클래스명만 변경.
- **변경 요약**:
  - 루트 패키지: `com.ktmmobile.mform` → `com.ktmmobile.msf`
  - 메인 클래스: `MformApiApplication` → `MsfApplication` (com.ktmmobile.msf)
  - 연동용 컨트롤러: `controller.v1.MformV1Controller` → `formComm.controller.FormCommRestController` (규칙: XXXRestController = 연동용)
  - 공통·M플랫폼: `common.*`, `config` → `msf.common.*`, `msf.common.config` (MplatFormService, MplatFormServerAdapter, vo, util, WebConfig 그대로 사용)
  - DTO: `dto.*` → `formComm.dto.*` (신청서 공통 API용 DTO)
- **유지**: MCP와 동일한 M플랫폼 연동 방식(juice.url, getParamMap, callService, CommonXmlVO 등)은 수정 없이 패키지 경로만 msf 로 변경.
- **참고**: `30.개발기능목록_담당.md`, `32.개발기능목록_담당_TOBE분석.md` 등에서 Controller/API 참조 시 `MformV1Controller` → `FormCommRestController`, `MplatFormService` → `MplatFormSvc`(com.ktmmobile.msf.common.mplatform) 로 갱신.
- **명명 규칙 적용 (서비스층)**: M플랫폼 연동 서비스는 규칙상 `서비스명칭Svc` → `MplatFormSvc` 로 변경 적용함.

## 개발 계획

### 1. 사업·구조 전제 (19~21줄 반영)

| 구분 | 기존 (M포탈) | 신규 (스마트서식지) |
|------|--------------|---------------------|
| **사용처** | M포탈 웹 | 대리점용 스마트서식지 |
| **역할** | 웹에서 신청서·서비스 처리 | M포탈과 **동일 기능** 제공 |
| **개발 방식** | 기존 M포탈 소스 | **신규 개발** (프론트 + **백엔드 모두 신규**) |
| **연동 API** | M플랫폼 ↔ KT(KOS) 연계 | **동일하게 구현** (M포탈 API 스펙·연동 방식 따름) |
| **화면** | M포탈 웹 UI | 화면은 **조금 다르게** 구성 (mform-web) |

- **정리**: 스마트서식지는 **프론트(mform-web)·백엔드 모두 신규로 개발**한다. 다만 **연동 API는 M포탈과 동일하게 구현**하고(요청/응답·M플랫폼·KT(KOS) 연동 스펙 동일), 기존 M포탈 코드를 재사용하는 것이 아니다.

```
[대리점] 스마트서식지 프론트 (mform-web, Vue3·JS)  ←  신규 개발
    │
    │  API 호출 (스펙은 M포탈과 동일)
    ▼
[스마트서식지 백엔드]  ←  신규 개발 (연동 API 동일하게 구현)
    │
    │  M플랫폼·KT(KOS) 연동 (M포탈과 동일 스펙/방식)
    ▼
[M플랫폼]
    │
    ▼
[KT(KOS)]
```

- **프론트·백엔드 모두 신규 개발.** M포탈 소스 재사용이 아님.
- **연동 API**: M포탈에서 쓰는 API 스펙·M플랫폼/KT(KOS) 연동 방식을 **동일하게 구현**하여 새 백엔드에서 제공.

---

### 3. 기술 스택 (mform-web 기준)

- **언어**: JavaScript (TypeScript 미사용)
- **프레임워크**: Vue 3, Vue Router 5, Pinia
- **빌드/스타일**: Vite 7, Tailwind CSS 4
- **기타**: reka-ui, @vueuse/core 등 (상세는 `doc/Mform-Web_폴더구조_분석.md` 참조)

---

### 4. 연동 API 동일 구현 원칙 (백엔드 신규 개발)

- 기능목록(`스마트서식지-AN-01-개발과제및기능목록_20260225.md`)의 **연동/API 컬럼**에 기재된 API를 기준으로 **스마트서식지 백엔드를 신규 개발**한다.
- **연동 API 스펙**은 M포탈과 **동일하게 구현** (요청/응답 형식, M플랫폼 호출, KT(KOS) 연동 방식 동일). 예: X01 가입정보조회, X20 부가서비스 조회, X21/X38 부가서비스 신청/해지, X19 요금상품 변경 등.
- 코드는 재사용이 아니라 **신규 작성**하며, M포탈 연동 명세·기능목록을 참고하여 **동일한 API·연동**을 새 백엔드에서 제공.

### 10. 공통·주의 사항

- **백엔드**: **신규 개발**한다. M포탈 모듈 재사용이 아니라, 스마트서식지 전용 백엔드를 새로 만들고 **연동 API는 M포탈과 동일하게 구현**한다.
- **연동 API 스펙**: 요청/응답·M플랫폼/KT(KOS) 연동 방식은 **M포탈에서 사용 중인 것과 동일**하게 구현 (참고: MCP 구조·`KT_M플랫폼_X78_X79_연동명세.md` 등).
- **uidoc 미비 시**: `스마트서식지-AN-01-개발과제및기능목록_20260225.md` 의 화면명·기능 설명·연동 API로 요구사항 보완.
