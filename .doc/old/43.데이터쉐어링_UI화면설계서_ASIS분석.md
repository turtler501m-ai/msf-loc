### 1.2 전체 플로우

```
[쉐어링(인증전)] mySharingCntrInfo
       │
       │ 로그인 또는 SMS 인증 후 "가입정보 확인"
       ▼
[쉐어링(인증후)] mySharingView  ← 회선 선택·조회
       │
       │ "데이터 쉐어링 결합하기" (결합 가능 시)
       ▼
[쉐어링 가입하기] reqSharingView  ← 본인인증·유심·약관·개통 요청
       │
       │ 본인인증 + 유심 유효성 + 약관 동의 후 "개통 요청"
       │ → saveDataSharingSimpleAjax (셀프개통) 후 insertOpenRequestAjax (X69/X70)
       ▼
[쉐어링 완료] reqSharingCompleteView
```

- **뒤로가기**: `history.pushState` + `onpopstate` 시 `/mySharingCntrInfo.do`로 이동하도록 처리됨.

## 2. AS-IS 분석 (기술 구조)

### 2.1 컨트롤러·URL 매핑

| URL | Method | Controller | 설명 |
|-----|--------|------------|------|
| `/mySharingCntrInfo.do`, `/m/mySharingCntrInfo.do` | GET | `MyShareDataController.doMySharingCntrInfo` | 쉐어링 인증전 화면. 로그인 시 바로 `mySharingView`로 redirect |
| `/content/mySharingView.do`, `/m/content/mySharingView.do` | GET/POST | `MyShareDataController.doMySharingView` | 쉐어링 인증후 화면. 회선 목록·요금제·X71 조회 결과 표시 |
| `/content/reqSharingView.do`, `/m/content/reqSharingView.do` | GET | `MyShareDataController.dorReqSharingView` | 쉐어링 가입하기 화면. `contractNum` 필수, 미입력 시 mySharingView redirect |
| `/content/insertOpenRequestAjax.do`, `/m/content/insertOpenRequestAjax.do` | POST | `MyShareDataController.doinsertOpenRequestAjax` | 데이터 쉐어링 처리 Ajax. X69 사전체크 후 X70 결합/해지 |
| `/content/reqSharingCompleteView.do`, `/m/content/reqSharingCompleteView.do` | GET/POST | `MyShareDataController.doReqSharingCompleteView` | 쉐어링 완료 화면. `session.opmdSvcNo` 없으면 mySharingView redirect |
| `/appform/saveDataSharingSimpleAjax.do`, `/m/appform/saveDataSharingSimpleAjax.do` | POST | `AppformController.saveDataSharingSimple` | 데이터 쉐어링 **신규 셀프개통** 요청 (유심·본인인증 기반, MCP_REQUEST 등 저장) |

- 세션 검사: `SessionChkInterceptor`에서 `/content/mySharingView.do`, `reqSharingView.do`, `reqSharingCompleteView.do` 등이 order URL 목록에 포함됨.

### 2.2 서비스·M-Platform 연동

**MyShareDataSvc / MyShareDataSvcImpl**

| 메서드 | M-Platform 이벤트 | 설명 |

- **X71** `mosharingList(custId, ncn, ctn)`

  - 응답: `MoscDataSharingResDto` – 가입 가능 대상 리스트, `rsltInd`(Y 등)로 결합 가능 여부 판단

### 2.5 데이터 흐름 요약

1. **인증전**  
   - 비로그인: SMS 인증 후 `menuType=sharing`으로 mySharingView 진입.  
   - 로그인: mySharingCntrInfo 접속 시 바로 mySharingView redirect.

2. **인증후(mySharingView)**  
   - 회선 선택 후 [조회]: mySharingView 재요청.  
   - 서버: `selectCntrList`/`selectCntrListNoLogin`, `selectSocDesc`, `selectCustomerType`, SHARINGRATE 코드로 요금제 쉐어링 가능 여부 판단, **X71** `mosharingList` 호출 → 결합 목록 또는 결합 가능 여부에 따라 분기 렌더링.

3. **가입하기(reqSharingView)**  
   - 진입: `contractNum`(ncn) 필수, 세션/회선 소유 검증.  
   - 개통 요청:  
     - **셀프개통**: `saveDataSharingSimpleAjax.do` (유심·본인인증 검증, MCP_REQUEST 저장, 스캔 전송 등) → 응답에 전화번호 등 반환.  
     - **쉐어링 결합**: `insertOpenRequestAjax.do`에서 `selfShareYn=Y`이면 X69 사전체크 후, `rsltInd=Y`인 대상 있으면 X70 결합 호출.  
   - 성공 시 `opmdSvcNo`를 세션에 넣고 reqSharingCompleteView로 이동.

4. **완료(reqSharingCompleteView)**  
   - 세션의 `opmdSvcNo`와 `ncn`으로 대표회선/데이터쉐어링 등록 회선 정보 조회(`selectSocDesc` 등) 후 표시.  
   - 확인 시 `opmdSvcNo` 세션 제거, mySharingCntrInfo로 이동.

### 2.6 기타 연동

- **데이터 쉐어링 개통 시 청구계정**: MCP-API `MypageController` 등에서 "데이터 쉐어링 개통시 청구계정 정보 조회" 사용.  
- **상품/약관**: `Constants.SHARINGRATE`(쉐어링 요금제 코드), `Constants.SHAR_INFO_CD`, `INFO_CD` 등으로 데이터쉐어링 소개·약관 노출.  
- **셀프개통 연동**: `appFormDesignView.js`에서 셀프개통인 경우 데이터 쉐어링 관련 UI 미노출 처리.

## 3. 정리

- **UI**: 인증전 → 인증후(회선 조회·결합 가능 여부) → 가입하기(본인인증·유심·약관·개통 요청) → 완료 4단계로 구성됨.  
- **AS-IS**: `MyShareDataController`가 쉐어링 전용 화면과 Ajax를 담당하고, M-Platform **X69(사전체크)·X70(결합/해지)·X71(결합 목록)** 호출은 `MyShareDataSvc` → `MplatFormService`에서 수행.  
- **셀프개통 경로**: 데이터쉐어링 신규 회선 개통은 `AppformController.saveDataSharingSimple`와 스캔/본인인증 연동 후, 동일 플로우에서 `insertOpenRequestAjax`로 X69/X70을 호출해 결합까지 처리함.

## 4. TO-BE 개발 시 이슈 사항

TO-BE 개발 시 아래 사항을 검토·정리하는 것을 권장한다.

### 4.1 버그·일관성

| 구분 | 내용 | 권장 조치 |
|------|------|-----------|
| **응답 키 오타** | 백엔드가 `REQUEST_KET`로 반환, 프론트는 동일 키로 수신. 전사적으로 `REQUEST_KEY` 사용 시 응답 키 통일 필요. | TO-BE에서 API 스펙을 `REQUEST_KEY`로 통일하고, 기존 `REQUEST_KET` 참조 제거. |
| **유심 입력 셀렉터** | `reqSharingView.js` 내 `$("#iccid")`(소문자)와 `$("#iccId")` 혼용. JSP는 `id="iccId"`. | 셀렉터를 `#iccId`로 통일(대소문자 일치). |
| **완료 화면 opmdSvcSocNm** | `doReqSharingCompleteView`에서 `opmdSvcNoContractNum`을 선언만 하고 할당하지 않아 항상 공백. `opmdSvcSocNm`이 세팅되지 않음. | 완료 화면에서 “데이터쉐어링 전용요금제”는 JSP에 하드코딩되어 있어 동작에는 문제 없으나, 실제 요금제명을 보여줄 경우 `opmdSvcNo` 기준 조회 로직·변수 할당 추가 검토. |

### 4.2 검증·플로우

| 구분 | 내용 | 권장 조치 |
|------|------|-----------|
| **필수 약관 검증** | `sharing_reg()` 안 `$('input[name=terms]').each()` 내부의 `return false`는 each 콜백만 종료하고 `sharing_reg()` 자체는 계속 진행됨. | 필수 약관 미동의 시 `sharing_reg()` 상단에서 조기 return 하도록 검증 로직 분리(예: 필수 항목 체크 후 하나라도 미체크면 alert 및 return). |
| **중복 요청 방지** | `mySharingView`에는 `overlapRequestCheck`가 있으나, **가입하기(reqSharingView)** 및 [개통 요청] Ajax에는 중복 클릭 방지가 없음. | 개통 요청 버튼 비활성화(또는 스피너) + 서버 측 동일 건 중복 처리 방지(트랜잭션/상태 체크) 적용 검토. |
| **에러 메시지** | 실패 시 `SIMPLE_OP2_MSG`, `RESULT_MSG` 등 세부 메시지 매핑이 주석 처리되어 있어, 사용자에게 “쉐어링 가입 실패했습니다.”만 노출됨. | M-Platform/OSST 응답 코드별 메시지 매핑 복구 및 프론트에 메시지 전달해, 사유별 안내 가능하도록 개선. |

### 4.3 세션·상태

| 구분 | 내용 | 권장 조치 |
|------|------|-----------|
| **완료 화면 직접 접근** | `reqSharingCompleteView`는 `session.opmdSvcNo` 없으면 redirect. 단, 새 탭/URL 직접 입력으로 진입 시 `myShareDataReqDto.ncn` 등이 비어 있을 수 있음. | GET 파라미터만으로 완료 화면 접근 시 필수 값 부재 처리(redirect 또는 에러 메시지) 명확히 정의. |

### 4.4 M-Platform·API

| 구분 | 내용 | 권장 조치 |
|------|------|-----------|
| **X69 crprCtn** | `MplatFormService.moscDataSharingChk`에서 `param.put("crprCtn", crprCtn)`가 주석 처리되어 있어, X69 요청에 crprCtn이 넘어가지 않음. 셀프개통 시에는 `insertOpenRequestAjax`에서 `crprCtn=""`로 호출. | X69 스펙 확인 후, 결합 대상 식별이 필요하면 crprCtn(또는 해당 파라미터) 전달 여부를 정리하고 필요 시 주석 해제 및 호출부 정리. |
| **해지(자회선 해지) UI** | X70은 `opmdWorkDivCd`로 결합(A)/해지(C) 구분. 현재 UI/문서에는 “이미 결합된 회선” 표시만 있고, **해지** 버튼/플로우는 없음. | 해지 기능 도입 시: 해지 가능 대상 조회, 확인 절차, X70 해지 호출, 완료 화면 설계 필요. |

### 4.5 아키텍처·역할 분리

| 구분 | 내용 | 권장 조치 |
|------|------|-----------|
| **이중 플로우** | 가입하기에서 ① `saveDataSharingSimpleAjax`(AppformController, 셀프개통·DB·스캔) → ② `insertOpenRequestAjax`(MyShareDataController, X69+X70) 순차 호출. 실패 시점에 따라 DB만 반영·M-Platform만 반영 등 불일치 가능. | TO-BE에서 “셀프개통 후 결합”을 하나의 트랜잭션/보상 단위로 설계할지, 또는 단계별 보상(롤백/재시도) 정책을 명시. |
| **Controller 분산** | 쉐어링 화면·Ajax는 `MyShareDataController`, 셀프개통 저장은 `AppformController`. | 유지보수성·테스트를 위해 쉐어링 전용 서비스/퍼사드로 흐름을 한곳에서 조율하는 구조 검토. |

### 4.7 기타

- **쉐어링 불가 요금제**: `SHARINGRATE` 코드로 관리. 요금제 추가/변경 시 코드 테이블 및 JSP 분기 반영 필요.
