# `/v1/addition/current` 연동 확인

**확인 대상**: `mformPost('/v1/addition/current', params)` 호출이 끝단까지 잘 연결되어 있는지

---

## 1. 프론트 (mform-web)

### 1.1 API 정의

| 항목 | 내용 | 상태 |
|------|------|------|
| 파일 | `src/api/serviceChange.js` | ✅ |
| 함수 | `getCurrentAddition(params)` | ✅ |
| 호출 | `return mformPost('/v1/addition/current', params)` | ✅ |
| params 스펙 | `{ ncn: string, ctn: string, custId?: string }` (JSDoc) | ✅ |

### 1.2 실제 요청

- **mform.js**: `mformPost(path, body)` → `fetch(BASE_URL + path, { method: 'POST', body: JSON.stringify(body) })`
- **BASE_URL**: `import.meta.env.VITE_API_BASE_URL || '/api'`
- **최종 URL**: `POST /api/v1/addition/current`
- **Body**: JSON `{ ncn, ctn, custId }` (전달한 params 그대로)

### 1.3 사용처

| 사용처 | API | 비고 |
|--------|-----|------|
| **IdentProd.vue** | ident.js 의 `getCurrentAddition` | `/v1/ident/addition/current` 호출 (별도 경로) |
| **서비스변경** (ChangeProd / McpAdditionEditPop) | serviceChange.js 의 `getCurrentAddition` | 정의만 있고 호출은 TODO (팝업 오픈 시 연동 예정) |

→ **serviceChange.js** 의 `mformPost('/v1/addition/current', params)` 는 **서비스변경 플로우에서 아직 호출되지 않고**, **명의변경**은 `/v1/ident/addition/current` 를 사용함.

---

## 2. 백엔드 (mform-api)

### 2.1 엔드포인트

| 항목 | 내용 | 상태 |
|------|------|------|
| 컨트롤러 | `MformV1Controller.java` | ✅ |
| 클래스 매핑 | `@RequestMapping("/api/v1")` | ✅ |
| 메서드 | `@PostMapping("/addition/current")` | ✅ |
| 최종 경로 | `POST /api/v1/addition/current` | ✅ |

### 2.2 요청/응답

| 항목 | 내용 | 상태 |
|------|------|------|
| 요청 DTO | `JoinInfoRequest` (ncn, ctn, custId) | ✅ |
| 응답 DTO | `AdditionCurrentResponse` (items) | ✅ |
| 현재 동작 | `res.setItems(Collections.emptyList())` → `{ "items": [] }` 반환 | ✅ |

### 2.3 명의변경용 경로

- `POST /api/v1/ident/addition/current` → `identAdditionCurrent()` → 내부에서 `additionCurrent(req)` 호출
- 요청/응답 스펙 동일. ✅

## 3. 연동 정합성

| 구간 | 확인 | 결과 |
|------|------|------|
| 프론트 URL | `/api` + `/v1/addition/current` = `/api/v1/addition/current` | ✅ |
| 백엔드 URL | `/api/v1` + `/addition/current` = `/api/v1/addition/current` | ✅ |
| 메서드 | POST | ✅ |
| 요청 Body | 프론트 `params` (ncn, ctn, custId) ↔ `JoinInfoRequest` | ✅ |
| 응답 | `AdditionCurrentResponse` → JSON `{ items: [] }` | ✅ |
| 프론트 응답 사용 | IdentProd: `data.items` 로 사용 (ident 경로 기준) | ✅ |

→ **`return mformPost('/v1/addition/current', params)` 자체는 잘 개발되어 있고**, 프론트 URL·백엔드 경로·요청/응답 형식이 서로 맞음.

---

## 4. 보완 권장 사항

1. **서비스변경에서 실제 호출**  
   - `McpAdditionEditPop` 열릴 때 `getCurrentAddition({ ncn, ctn, custId })` (serviceChange.js) 호출해 목록 갱신하는 부분이 아직 TODO.  
   - 위 API가 서비스변경 플로우에서도 쓰이려면 해당 TODO 구현 필요.

2. **params 전달**  
   - X20 스펙상 ncn, ctn, custId 모두 사용.  
   - 호출하는 쪽에서 **가능하면 `{ ncn, ctn, custId }` 를 모두 넘기도록** 하는 것이 좋음 (지금 IdentProd는 `{ ctn }` 만 넘김).

3. **응답 items 형식 (선택)**  
   - 현재 백엔드는 `List<Object>`, Mock은 빈 리스트.  
   - 실연동/목업 데이터 넣을 때는 M포털 X20 응답과 맞추어 `soc`, `socDescription`, `socRateValue` 등 필드 정의해 두면 프론트 처리하기 쉬움.

---

## 5. 결론

- **`return mformPost('/v1/addition/current', params)`**  
  - 경로·메서드·요청/응답 구조가 백엔드와 일치하고,  
  - 백엔드도 `/api/v1/addition/current` 로 정상 수신·응답하도록 되어 있어 **잘 개발된 상태**로 볼 수 있음.
- 다만 **서비스변경**에서는 이 API를 아직 호출하지 않고 있으며, **명의변경**은 `/v1/ident/addition/current` 를 사용 중이므로,  
  서비스변경 플로우에서 “현재 부가 목록”을 쓰려면 **McpAdditionEditPop 등에서 위 `getCurrentAddition` 을 호출**하도록 연결만 보완하면 됨.
