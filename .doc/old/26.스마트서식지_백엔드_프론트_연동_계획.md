# 스마트서식지 백엔드·프론트 연동 계획

기존 MCP 소스 및 ktm/doc 문서를 참조하여 스마트서식지(mform-web) 전용 백엔드를 구현하고, 기존 프론트 로직을 신규 개발에 적용하기 위한 계획입니다.

---

## 1. 참조 소스·문서

| 구분 | 경로 | 용도 |
|------|------|------|
| M플랫폼 X01 | mcp/mcp-portal-was/.../MplatFormService.java (perMyktfInfo) | 가입정보 조회 스펙 |
| X01 VO | mcp/mcp-portal-was/.../vo/MpPerMyktfInfoVO.java | 응답 필드: addr, email, homeTel, initActivationDate |
| 포털 컨트롤러 | mcp/mcp-portal-was, mcp/mcp-api appform·mypage | API 패턴 참고 |
| 문서 | ktm/doc/개발기능목록_담당.md, M포털_변경처리_프로세스_분석.md, 스마트서식지_개발플랜.md | API·화면 매핑 |

---

## 2. 아키텍처

```
[mform-web (Vue3 SPA)]
    │  VITE_API_BASE_URL → http://localhost:8081/api
    │  POST /api/v1/join-info, /api/v1/addition/current ...
    ▼
[mform-api (Spring Boot)]  ← 신규 구현
    │  (1차) Mock 응답으로 테스트
    │  (2차) MCP M플랫폼 연동 또는 mcp-api 프록시
    ▼
[M플랫폼 / mcp-api]  ← 기존 (연동 시)
```

- **원칙**: mform-web이 호출하는 API 스펙은 M포털(X01 등)과 동일하게 유지. 코드 재사용이 아닌 **동일 스펙** 구현.

---

## 3. 백엔드 구현 범위 (1차)

### 3.1 프로젝트

- **이름**: mform-api
- **위치**: workspace/mform-api
- **기술**: Spring Boot 2.7, Java 11, REST (JSON)

### 3.2 API 엔드포인트 (프론트 api/*.js와 1:1)

| 메서드 | 경로 | 요청 body | 응답 (성공) | 참고 MCP |
|--------|------|------------|-------------|----------|
| POST | /api/v1/join-info | `{ ncn?, ctn, custId? }` | `{ success, addr, email, homeTel, initActivationDate }` | X01 perMyktfInfo |
| POST | /api/v1/service-change/concurrent-check | `{ selectedCodes }` | `{ valid }` | 동시변경 불가 체크 |
| POST | /api/v1/addition/current | `{ ncn, ctn, custId? }` | `{ items }` | X20 |
| POST | /api/v1/ident/join-info | 동일 join-info 스펙 | 동일 | X01 |
| POST | /api/v1/cancel/join-info | 동일 | 동일 | X01 |

### 3.3 1차 구현 내용

- **join-info**: ctn(휴대폰번호) 필수. 성공 시 X01 형식 JSON 반환 (Mock: 고정 샘플 데이터). 실패 시 4xx + message.
- **concurrent-check**: selectedCodes 검증 후 valid: true/false (Mock).
- **addition/current**: Mock 빈 배열 또는 샘플 데이터.
- **CORS**: mform-web 개발 서버(예: http://localhost:5173) 허용.

### 3.4 2차 (추후)

- mform-api에서 MCP MplatFormService 호출 (동일 네트워크/배포 시) 또는 mcp-api HTTP 클라이언트로 프록시.
- 스마트서식지 DB·신청서 등록·이미징·SMS/카톡 연동.

## 4. 프론트 엔드 적용 로직 (기존 MCP 로직 반영)

### 4.1 휴대폰 번호 인증 (X01)

- **기존 MCP**: MyinfoController 등에서 `perMyktfInfo(ncn, ctn, custId)` 호출 후 `addr`, `email`, `initActivationDate` 등 화면에 반영.
- **mform-web 적용**:
  - `getJoinInfo({ ncn, ctn, custId })` 호출 후 성공 시 응답의 `addr`, `email`, `homeTel`을 고객 폼(주소, 이메일)에 자동 반영.
  - 실패 시 인증 완료 처리하지 않고 에러 메시지 표시.

### 4.2 서비스변경 동시 처리 불가

- **기존 MCP**: conPreCheck 등으로 번호변경·분실복구·요금상품변경·핸드폰기변 2개 이상 선택 시 제한.
- **mform-web**: 이미 ChangeType.vue에서 CORE_CHANGE 상수로 동일 로직 적용. 백엔드 `concurrent-check` 연동 시 선택 목록 전달해 검증 강화 가능.

### 4.3 부가서비스 조회 (X20)

- **기존 MCP**: moscCombInfo(X20) 호출 후 현재 부가 목록 표시.
- **mform-web**: ChangeProd.vue에서 `getCurrentAddition` 호출 후 연동 시 목록 표시. 1차는 Mock으로 다음 단계 활성화만 유지.
