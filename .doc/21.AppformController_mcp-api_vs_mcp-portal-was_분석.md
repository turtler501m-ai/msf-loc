## 2. 아키텍처 비교

### mcp-api — REST API 레이어

```
Controller → Mapper(직접) → DB
```

- **목적**: 외부 클라이언트(서식지 앱 등)에 데이터를 JSON으로 제공하는 단순 API
- **계층**: Controller → Mapper 직접 호출 (Service 계층 없음)
- **상태**: Stateless (Session 없음)
- **주입 필드**: `AppformMapper` 1개만 주입

### mcp-portal-was — 엔터프라이즈 MVC

```
Controller → Service → Mapper → DB
                     → MplatFormService → KT(KOS)
```

- **목적**: M포탈 전체 비즈니스 로직 처리 (웹+Ajax 하이브리드)
- **계층**: Controller → Service(비즈니스 로직) → Mapper
- **상태**: Stateful (Session으로 인증 상태, 임시 데이터 관리)
- **주입 필드**: 20개 이상 (Service, 설정값, 외부 연동)

## 5. 동일 URL의 구현 차이 — selRMemberAjax

두 모듈 모두 존재하는 대표 URL을 통해 구현 수준 차이를 확인할 수 있다.

### mcp-api `/appform/selRMemberAjax`

```java
@RequestMapping(value = "/appform/selRMemberAjax", method = RequestMethod.POST)
public JuoSubInfoDto selRMemberAjax(@RequestBody JuoSubInfoDto juoSubInfoDto) {
    JuoSubInfoDto result = null;
    try {
        result = appformMapper.selectSelRMemberAjax(juoSubInfoDto);
    } catch(Exception e) {
        throw new McpCommonJsonException(COMMON_EXCEPTION);
    }
    return result;  // ← DB 결과 그대로 반환
}
```

**처리 수준**: DB 조회 → 결과 반환 (4줄 로직)

### mcp-portal-was `/appform/selRMemberAjax.do`

```
1. 주민번호 입력값 검증 (blank 체크)
2. AppformSvc.selRMemberAjax() 호출
3. ACE-256 암호화된 주민번호 복호화
4. 입력 생년월일 vs DB 주민번호 앞 7자리 비교
5. 인가된 사용자 체크 (MypageService.checkAuthUser)
6. 기기변경 가능여부 판단 (DvcChgYn: Y/E/N)
7. 외국인/미성년자 고객 유형 분류 (주민번호 성별코드 분석)
8. 연령 계산 (19세 미만 미성년자 처리)
9. NICE 인증 Step 정보 검증 (certService.vdlCertInfo)
10. Session에 인증 결과 저장 (SessionUtils.saveChangeAut)
11. 상태코드 분기 반환 (AJAX_SUCCESS / 0002 / 0003 / 0004)
```

**처리 수준**: 다단계 검증 + 암호화 + 인증 + Session (100줄+ 로직)

```java
// 신청서 임시저장 (updateRequestTemp.do) - 핵심 로직 요약
@RequestMapping(value = "/appForm/updateRequestTemp.do")
@ResponseBody
public Boolean updateRequestTemp(AppformReqDto dto) {
    // 1. 요청키 유효성 검증
    if (0 >= dto.getRequestKey()) throw new McpCommonJsonException(...);

    // 2. NICE 인증 완료 여부 확인
    NiceResDto sessNiceRes = SessionUtils.getNiceResCookieBean();
    if (sessNiceRes == null) {
        // SMS 인증 대체 처리
        if ("01".equals(dto.getTmpStep()) && "Y".equals(dto.getTelAdvice())) {
            if (0 >= certService.getModuTypeStepCnt("smsAuth", ""))
                certService.delStepInfo(1);
        }
        throw new McpCommonJsonException("0003", NICE_CERT_EXCEPTION);
    }

    // 3. Step별 청소년/외국인 분기 처리
    if ("01".equals(dto.getTmpStep())) {
        if (CSTMR_TYPE_NM.equals(dto.getCstmrType())) {
            // 청소년 본인정보 추가 검증
            certService.vdlCertInfo("D", certKey, certValue);
        }
    }

    // 4. Service 위임
    return appformSvc.updateRequestTempStep1(dto);
}
```

## 8. 발견된 코드 이슈 (mcp-api)

### 메서드명 오류 — insertDisPrmtApd (라인 471)

```java
// URL: /appform/insertDisPrmtApd
@RequestMapping(value = "/appform/insertDisPrmtApd", method = RequestMethod.POST)
public int chkCstmrInfo(@RequestBody AppformReqDto disApdParam) {  // ⚠️ 메서드명이 chkCstmrInfo
    return appformMapper.insertDisPrmtApd(disApdParam);  // 실제 기능은 프로모션 적용

```

→ 실제 기능(프로모션 적용)과 메서드명(친구초대 확인)이 불일치. TOBE 구현 시 혼동 주의.

### 중복 API — 보험 조회 2개

```java
// 완전히 동일한 Mapper를 호출하는 두 URL
```

### 핵심 결론

- **mcp-api** = 데이터 조회/저장 전용 REST API (MSF의 현재 수준과 유사)
- **mcp-portal-was** = 실제 사용자 서비스 처리 엔드포인트 (인증·검증·M플랫폼 연동 포함)
- MSF 개발 시 **기능별 ASIS 참조는 반드시 mcp-portal-was의 AppformController** 기준으로 해야 하며, mcp-api는 DB 쿼리 파라미터 구조 파악 용도로만 활용.
