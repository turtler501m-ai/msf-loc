# 스마트서식지 서비스해지 처리 연동 설계서

> 버전: v1.3 / 작성일: 2026-03-25 / 수정일: 2026-03-25
> 관련 REQ: REQ_010401, REQ_010402
> 관련 기능번호: C124~C129, C132~C136

---

## 1. 개요 및 요구사항

### 1.1 목적

스마트서식지(MSF) App에서 접수된 서비스해지 신청을 MSP(MVNO Sales Portal) 해지상담신청 처리 화면에서
조회·처리할 수 있도록 연동하는 설계.
M플랫폼 직접 전송 방식이 아닌 **M포탈 테이블 적재 방식**으로 처리.

> **M포탈 DB DDL 변경 없음** — 기존 컬럼(REGST_ID, ONLINE_AUTH_INFO) 활용으로 MSF 건 식별.

### 1.2 요구사항 정의

| # | 요구사항 | 관련 컴포넌트 |
|---|---------|-------------|
| R1 | MSF App 신청 → M포탈 API → MCP_CANCEL_REQUEST 적재 (기존 컬럼 활용) | MSF apply API + M포탈 적재 API |
| R2 | 신청서 이미지를 MSP에서 열람 가능하도록 SCAN 서버 전송 + MCP_CANCEL_REQUEST.SCAN_ID 동기화 | FormSend (reqType=CC) + M포탈 scan-update API |
| R3 | MSP 해지상담신청 화면에서 MSF 건 식별 및 조회·처리 분기 | MSP CanCustMgmt_SQL.xml 수정 |
| R4 | MSP 처리 완료 후 MSF_REQUEST_CANCEL.PROC_CD 배치 동기화 | MSF 배치 |
| R5 | 서비스해지 완료에 따른 M포탈 계약정보 영향도 체크 | 영향도 분석 |

```

SvcCnclController.apply()  ─── 기구현 (MSF_REQUEST_CANCEL INSERT, PROC_CD='RC')

    │
    ▼

    │     REGST_ID='MFORM' → SOURCE_CD='MSF' 로 표시
    │     ONLINE_AUTH_INFO → MSF REQUEST_KEY 포함
    │     MSP_REQUEST_DTL C → LEFT OUTER JOIN 으로 변경
    │
    ├─ 처리: updateCanCsl.json — 변경없음
    │     → PROC_CD 업데이트 (MCP_CANCEL_REQUEST@DL_MCP)
    │     → SCAN 합본 처리 (MSF 건도 동일 패턴)
    │
    └─ MSF 건 처리완료 시 — MSP에서 MSF API 호출
          POST /msf/api/v1/cancel/proc → MSF_REQUEST_CANCEL.PROC_CD='CP' 업데이트
          ※ 또는 R4 배치로 대체

[MSF 배치]  ─── [R4] 신규 배치
    └─ MSF_REQUEST_CANCEL WHERE PROC_CD='RC' AND RES_NO IS NOT NULL 조회
```

ㅇ MSF App 신청 접수
  - 신청서 저장 (MSF_REQUEST_CANCEL, PROC_CD='RC')                                                                                                                                                                                                             
  - M포탈에 해지 건 적재 → custReqSeq 받아 RES_NO에 저장                                                                                                                                                                                                     
  - 신청서 이미지 SCAN 서버 전송 → M포탈 SCAN_ID 동기화 (MSP 합본 처리 대비)

  - MSF 건 처리완료 시 — MSP에서 MSF API 호출

ㅇ MSF 완료 동기화 (배치)
  - 주기적으로 M포탈 상태 조회 → MSP에서 처리 완료된 건(PROC_CD='CP')을 MSF에 반영

## 4. 상세 설계

### 4.1 [R1] M포탈 테이블 데이터 적재

#### 4.1.1 MSF 건 식별 방법 (M포탈 DB DDL 변경 없음)

`MCP_CANCEL_REQUEST`의 **기존 컬럼**을 활용하여 MSF 건을 식별한다.

| 컬럼 | ASIS 용도 | MSF 활용 값 |
|------|---------|-----------|
| `REGST_ID` | 등록자 사용자 ID | `'MFORM'` 고정 — MSF 건 식별자 |
| `ONLINE_AUTH_INFO` | NICE 인증 ReqNo/ResNo | `'MSF:{requestKey}'` — MSF REQUEST_KEY 저장 |

- MSP에서 `REGST_ID = 'MFORM'` 조건으로 MSF 건 식별
- `ONLINE_AUTH_INFO`에서 `'MSF:'` 이후 값을 파싱하면 MSF REQUEST_KEY 추출 가능

#### 4.1.2 M포탈 적재 API (MCP 신규 개발)

```
POST /mcp/api/cancel/msf-register

요청 바디:
{
  "contractNum"    : "123456789",
  "cancelMobileNo" : "01012345678",
  "cstmrName"      : "홍길동",
  "cstmrType"      : "IC",
  "receiveMobileNo": "01012345678",
  "receiveWayCd"   : "P",
  "regstId"        : "MFORM",    ← MSF 건 식별자 (고정)
  "sourceKey"      : "123",      ← MSF REQUEST_KEY (Long → String)
  "payAmt"         : 10000,
  "pnltAmt"        : 5000,
  "lastSumAmt"     : 15000
}
※ M포탈에서 ONLINE_AUTH_INFO = 'MSF:' + sourceKey 형태로 조합하여 저장

M포탈 INSERT:
  NMCP_CUST_REQUEST_MST: REQ_TYPE='CC', USERID='MFORM', ...
  MCP_CANCEL_REQUEST:    REGST_ID='MFORM', ONLINE_AUTH_INFO='MSF:{sourceKey}',
                         PROC_CD='RC', SCAN_ID=NULL (이후 scan-update API로 갱신)

응답:
{
  "resultCode" : "00000",
  "custReqSeq" : "2026032500001",    ← MSF_REQUEST_CANCEL.RES_NO에 저장
  "message"    : ""
}
```

#### 4.1.3 MSF SvcCnclSvcImpl.apply() 처리 순서

```
1. [기구현] 유효성 검증
2. [기구현] 중복접수 체크
3. [기구현] MSF_REQUEST_CANCEL INSERT → requestKey 획득
4. [R1] M포탈 msf-register API 호출
   - 성공: MSF_REQUEST_CANCEL.RES_NO = custReqSeq UPDATE
   - 실패: warn 로깅 후 계속 진행 (비치명적)
5. [R2-①] FormCommRestSvc.sendScan(requestKey, "CC")
   - 성공: MSF_REQUEST_CANCEL.SCAN_ID / APP_FORM_YN='Y' UPDATE
          → M포탈 scan-update API 호출 (RES_NO + scanId 전달)
   - 실패: warn 로깅 후 계속 진행 (비치명적)
6. [기구현] requestKey 반환
```

### 4.2 [R2] SCAN 서버 전송 및 MSP 합본 연동

#### 4.2.1 SCAN 서버 전송 (기구현)

`FormCommRestSvc.sendScan()` reqType=CC 로 MSF_REQUEST_CANCEL 기반 XML 생성·전송.

| XML 항목 | MSF_REQUEST_CANCEL 컬럼 |
|---------|------------------------|
| 고객명 | `CSTMR_NM` |
| 해지 번호 | `CANCEL_MOBILE_NO` |
| 계약번호 | `CONTRACT_NUM` |
| 잔여요금 | `PAY_AMT` |
| 위약금 | `PNLT_AMT` |
| 최종정산금 | `LAST_SUM_AMT` |
| 해지 후 이용 | `CANCEL_USE_COMPANY_CD` |
| 신청일 | `REG_DT` |
| 대리점코드 | `AGENT_CD` |

#### 4.2.2 MCP_CANCEL_REQUEST.SCAN_ID 동기화 (신규 M포탈 API)

MSP의 `updateCanCsl` 합본 처리는 `MCP_CANCEL_REQUEST.SCAN_ID`를 기준으로 동작한다.
MSF SCAN 전송 후 M포탈 API를 호출해 해당 컬럼을 갱신한다.

```
POST /mcp/api/cancel/update-scan

요청:
{
  "custReqSeq" : "2026032500001",    ← RES_NO (msf-register 응답값)
  "scanId"     : "MOCK_CC_123"       ← SCAN 서버 전송 결과 scanId
}

M포탈 처리:
  UPDATE MCP_CANCEL_REQUEST
     SET SCAN_ID = #{scanId}
   WHERE CUST_REQ_SEQ = #{custReqSeq}

응답:
{ "resultCode": "00000" }
```

### 4.3 [R3] MSP 해지상담신청 화면 분기 처리

> **개발 주체: MSP 팀**
> 대상 파일: `ktisMvno/src/main/resources/egovframework/sqlmap/rcp/canCustMgmt/CanCustMgmt_SQL.xml`

#### 4.3.1 현행 SQL 문제점

```sql
-- 현행: MSP_REQUEST_DTL INNER JOIN → MSF 건(데이터 없는 경우) 목록 미표시 위험
FROM MCP_CANCEL_REQUEST@DL_MCP A, MSP_JUO_SUB_INFO B, MSP_REQUEST_DTL C
WHERE A.CONTRACT_NUM = B.CONTRACT_NUM
  AND B.CONTRACT_NUM = C.CONTRACT_NUM   ← INNER JOIN

-- MSF 건은 REGST_ID='MFORM' 이지만 SOURCE 컬럼이 없어 구분 불가
-- SCAN_ID 합본 시 NEW_SCAN_ID(A.SCAN_ID) = NULL → 합본 처리 불가
```

#### 4.3.2 수정 내용: selectCanCslListInfo

```sql
<sql id="selectCanCslListInfo">
    SELECT
        A.CUST_REQ_SEQ
        , A.CONTRACT_NUM
        , B.SUB_LINK_NAME AS CSTMR_NAME
        , SUBSTR(CRYPTO_AES256.DEC_AES(B.USER_SSN), 1, 6) AS BIRTH
        , CASE WHEN C.CSTMR_TYPE = 'JP' OR C.CSTMR_TYPE = 'PP' THEN ''
               ELSE SUBSTR(REPLACE(CRYPTO_AES256.DEC_AES(B.USER_SSN),'-',''), 7, 1)
          END GENDER
        , A.CANCEL_MOBILE_NO
        , A.RECEIVE_MOBILE_NO
        , C.LST_RATE_CD                         -- MSF 건은 NULL 가능 (LEFT JOIN)
        , C.LST_RATE_NM                         -- MSF 건은 NULL 가능 (LEFT JOIN)
        , B.MODEL_NAME AS LST_MODEL_NM
        , B.INTM_SRL_NO AS LST_MODEL_SRL_NUM
        /* ... 기존 컬럼 유지 ... */
        , A.PROC_CD
        , (SELECT DTL_CD_NM FROM NMCP_CD_DTL@DL_MCP
           WHERE CD_GROUP_ID = 'CL05' AND DTL_CD = A.PROC_CD) AS PROC_CD_NM
        , A.MEMO
        , C.RES_NO
        , A.SCAN_ID  AS NEW_SCAN_ID
        , B.SCAN_ID  AS OLD_SCAN_ID
        /* ↓ 추가 컬럼 */
        , CASE WHEN A.REGST_ID = 'MFORM' THEN 'MSF' ELSE 'MCP' END AS SOURCE_CD
        , A.ONLINE_AUTH_INFO    -- 'MSF:{requestKey}' 파싱용 (SOURCE_CD='MSF'일 때)
      FROM MCP_CANCEL_REQUEST@DL_MCP A
           JOIN MSP_JUO_SUB_INFO B ON A.CONTRACT_NUM = B.CONTRACT_NUM
           LEFT OUTER JOIN MSP_REQUEST_DTL C ON B.CONTRACT_NUM = C.CONTRACT_NUM  -- ← INNER→LEFT 변경
     WHERE 1 = 1
    /* ... 기존 검색 조건 유지 ... */
    ORDER BY A.REGST_DTTM DESC
</sql>
```

#### 4.3.3 수정 내용: updateCanCslProcCd — 변경 없음

MSF 건도 `MCP_CANCEL_REQUEST@DL_MCP` PROC_CD 업데이트 동일 처리.

```sql
UPDATE MCP_CANCEL_REQUEST@DL_MCP
   SET MEMO      = #{memo}
     , PROC_CD   = #{procCd}
     , RVISN_ID  = #{sessionUserId}
     , RVISN_DTTM = SYSDATE
 WHERE CUST_REQ_SEQ = #{custReqSeq}
   AND PROC_CD != 'CP'
```

#### 4.3.4 수정 내용: updateCanCsl 합본 처리 — 변경 없음

- `NEW_SCAN_ID` = `A.SCAN_ID` (MSF가 SCAN 전송 후 M포탈 scan-update API로 저장한 값)
- `OLD_SCAN_ID` = `B.SCAN_ID` (MSP 기존 서식지)
- 동일 패턴으로 EMV_SCAN_FILE@DL_MCP 합본 처리

#### 4.3.5 화면 표시 가이드 (MSP 팀 참고)

| 항목 | MCP 건 | MSF 건 |
|------|-------|-------|
| SOURCE_CD | 'MCP' | 'MSF' |
| 고객명 | MSP_JUO_SUB_INFO.SUB_LINK_NAME | 동일 |
| 요금제 | MSP_REQUEST_DTL.LST_RATE_NM | NULL 가능 |
| MSF 신청 키 | — | ONLINE_AUTH_INFO에서 'MSF:' 이후 파싱 |
| 합본 처리 | 기존 동일 | MCP_CANCEL_REQUEST.SCAN_ID 채워진 후 동일 |

### 4.4 [R4] 처리 완료 배치 동기화

#### 4.4.1 개요

MSP 담당자가 해지 처리를 완료(`MCP_CANCEL_REQUEST.PROC_CD='CP'`)하면
M플랫폼을 통해 KT 실시스템에서 서비스해지가 실제로 처리된다.
MSF 스마트신청서 App은 주기적인 배치로 이 완료 결과를 인지하여
`MSF_REQUEST_CANCEL.PROC_CD`를 동기화한다.

```
MSF App 신청 → M포탈 적재 → MSP 담당자 해지 처리
                                    ↓
                        [MCP_CANCEL_REQUEST.PROC_CD='CP']
                        [M플랫폼 → KT 실시스템 해지 완료]
                        [M전산 계약상태 = 해지(C)]
                                    ↓
                        ┌─────────────────────────────┐
                        │  SvcCnclSyncBatch (MSF)     │
                        │  6시간 주기 실행             │
                        └─────────────────────────────┘
                                    ↓
                   [MSF_REQUEST_CANCEL.PROC_CD = 'CP' 동기화]
```

> **처리 완료 확인 방안 2가지** (아래 4.4.3, 4.4.4 참조)
> Phase 2 이전에는 **방안 B** (M플랫폼 X01 직접 조회)로 선행 구현.
> Phase 2 완료 후 **방안 A** (M포탈 status-batch)로 전환 또는 병행.

---

#### 4.4.2 배치 공통 스펙

| 항목 | 내용 |
|------|------|
| 배치명 | `SvcCnclSyncBatch` |
| 실행 주기 | `0 0 */6 * * *` (6시간마다) |
| 처리 대상 | `MSF_REQUEST_CANCEL WHERE PROC_CD='RC'` |
| 처리 단위 | 건별 순차 처리 (100건 초과 시 페이지 분할) |
| 오류 처리 | 건별 try-catch — 오류 건 warn 로깅 후 다음 건 계속 |
| 완료 조건 | `PROC_CD='CP'` 업데이트 후 배치 대상에서 자동 제외 |

> **⚠ MSF_REQUEST_CANCEL.PROC_DT 컬럼 미존재** — 처리일 기록이 필요하면
> `ALTER TABLE MSF_REQUEST_CANCEL ADD PROC_DT VARCHAR(8)` 컬럼 추가 협의 필요.
> 협의 전까지는 `PROC_CD` 업데이트만 수행.

#### 4.4.3 방안 A — M포탈 status-batch API 조회 *(Phase 2 이후 적용)*

> 전제: M포탈팀이 `POST /mcp/api/cancel/status-batch` 개발 완료 후.
> `RES_NO(custReqSeq)` 보유 건에 대해 M포탈 처리 상태를 일괄 조회.

```
SvcCnclSyncBatch.runByMcpApi()
    │
    ├─ 1. SELECT REQUEST_KEY, RES_NO
    │       FROM MSF_REQUEST_CANCEL
    │      WHERE PROC_CD = 'RC'
    │        AND RES_NO IS NOT NULL
    │
    ├─ 2. POST /mcp/api/cancel/status-batch
    │       요청: { "custReqSeqs": ["2026032500001", ...] }
    │       응답: [{ "custReqSeq": "...", "procCd": "CP"/"RC", "procDt": "YYYYMMDD" }, ...]
    │
    └─ 3. PROC_CD='CP' 건:
           UPDATE MSF_REQUEST_CANCEL
              SET PROC_CD = 'CP'
           -- PROC_DT = #{procDt}   ← 컬럼 추가 협의 후 활성화
            WHERE RES_NO  = #{custReqSeq}
              AND PROC_CD = 'RC'    ← 이중 업데이트 방지
```

**M포탈 신규 API 스펙**

```
POST /mcp/api/cancel/status-batch

요청:
{ "custReqSeqs": ["2026032500001", "2026032500002"] }

응답:
{
  "resultCode": "00000",
  "items": [
    { "custReqSeq": "2026032500001", "procCd": "CP", "procDt": "20260325" },
    { "custReqSeq": "2026032500002", "procCd": "RC", "procDt": null }
  ]
}
```

#### 4.4.4 방안 B — M플랫폼 X01 계약상태 직접 조회 *(Phase 1 선행 구현)*

> 전제: M포탈 API 개발 전에도 MSF 단독 구현 가능.
> M플랫폼 X01(perMyktfInfo)으로 계약상태를 조회하여 해지 완료 여부를 판정.
> `RES_NO` 없는 건(M포탈 적재 실패)도 처리 가능.

```
SvcCnclSyncBatch.runByMplatform()
    │
    ├─ 1. SELECT REQUEST_KEY, CANCEL_MOBILE_NO, CONTRACT_NUM
    │       FROM MSF_REQUEST_CANCEL
    │      WHERE PROC_CD = 'RC'
    │
    ├─ 2. 건별 M플랫폼 X01 조회
    │       MplatFormSvc.perMyktfInfo(contractNum, cancelMobileNo, null)
    │         → 응답: ACNT_STAT (계약상태코드)
    │
    ├─ 3. 해지 완료 판정
    │       ACNT_STAT = 'C'     → 해지 완료 → 4번으로
    │       ACNT_STAT = 'A'/'S' → 처리 중   → 스킵
    │       X01 실패 / 계약 없음 → warn 로깅 → 스킵
    │
    └─ 4. 해지 완료 건:
           UPDATE MSF_REQUEST_CANCEL
              SET PROC_CD = 'CP'
           -- PROC_DT = TO_CHAR(NOW(), 'YYYYMMDD')  ← 컬럼 추가 협의 후 활성화
            WHERE REQUEST_KEY = #{requestKey}
              AND PROC_CD     = 'RC'
```

**X01 계약상태코드(ACNT_STAT) 판정 기준**

| ACNT_STAT | 의미 | 배치 처리 |
|-----------|------|----------|
| `C` | 해지 완료 | `PROC_CD='CP'` 업데이트 |
| `A` | 정상 사용 중 | 스킵 (처리 미완료) |
| `S` | 일시정지 | 스킵 |
| 응답 없음/오류 | X01 조회 실패 | warn 로깅 후 스킵 |

> ※ ACNT_STAT 실제 코드값은 M플랫폼 X01 응답 스펙 확인 후 보완 필요.

#### 4.4.5 방안 선택 기준 및 전환 전략

| 구분 | 방안 A (M포탈 API) | 방안 B (M플랫폼 X01) |
|------|------------------|---------------------|
| 구현 가능 시점 | M포탈팀 API 개발 완료 후 | MSF 단독 즉시 구현 |
| 완료 기준 | MSP 처리 완료(PROC_CD='CP') | KT 실시스템 계약상태 해지 |
| 배치 부하 | 1회 bulk API (경량) | 건별 M플랫폼 호출 (부하 있음) |
| RES_NO 없는 건 처리 | 불가 | 가능 |

**단계별 전환:**

```
Phase 1 현재 → 방안 B: M플랫폼 X01 직접 조회 (MSF 단독 구현)
Phase 2 완료 → 방안 A 추가:
               RES_NO 있는 건 → 방안 A (M포탈 status-batch, 경량)
               RES_NO 없는 건 → 방안 B 폴백 (M플랫폼 X01)
```

### 4.5 [R5] 서비스해지 완료 영향도

| 영향 영역 | 변경 내용 | MSF 조치 |
|---------|---------|---------|
| M포탈 계약 상태 | 해지완료 → 계약상태=해지(C) | X01 재조회 시 자동 반영 |
| MSF_REQUEST_CANCEL.PROC_CD | RC → CP | R4 배치 처리 |
| MSF 재신청 방지 | CP 건 중복체크 | `PROC_CD != 'BK'` 조건으로 CP도 차단됨 (현행 유지) |
| 고객 계약조회 | 해지 후 계약 없음 | M전산 DB 업데이트 후 자연 반영 |

```
타임라인:
T0 — MSF App 신청 (MSF_REQUEST_CANCEL PROC_CD='RC')
T1 — M포탈 적재 (MCP_CANCEL_REQUEST PROC_CD='RC', REGST_ID='MFORM')
T2 — MSP 담당자 해지 처리 → MCP_CANCEL_REQUEST.PROC_CD='CP'
T3 — MSF 배치 (6시간 주기) → MSF_REQUEST_CANCEL.PROC_CD='CP' 동기화
```

## 5. DB 스키마 변경 계획

### 5.1 MSF_REQUEST_CANCEL — 변경 없음

기존 `RES_NO` 컬럼을 M포탈 custReqSeq 저장용으로 확정 사용.

| 컬럼 | 용도 |
|------|------|
| `RES_NO` | M포탈 msf-register 응답 custReqSeq 저장 |
| `SCAN_ID` | SCAN 서버 전송 결과 scanId 저장 (기구현) |
| `PROC_CD` | RC(접수) → CP(배치 동기화 후) |

### 5.2 MCP_CANCEL_REQUEST — DDL 변경 없음

기존 컬럼 활용으로 DDL 추가 불필요.

| 컬럼 | MSF 활용 값 |
|------|-----------|
| `REGST_ID` | `'MFORM'` — MSF 건 식별자 |
| `ONLINE_AUTH_INFO` | `'MSF:{requestKey}'` — MSF REQUEST_KEY |
| `SCAN_ID` | MSF SCAN 전송 후 scan-update API로 갱신 |
