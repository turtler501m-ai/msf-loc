## 1. 업무 개요

MSP(MVNO Sales Portal) 담당자(본사 직원)가 고객의 해지상담 신청 건을 조회하고 처리 상태를 갱신하는 후선 업무.  
MCP 포탈에서 고객이 신청한 해지 건(`MCP_CANCEL_REQUEST@DL_MCP`)을 DB Link로 직접 읽어 처리한다.

## 3. 목록 조회 SQL 분석 (`selectCanCslListInfo`)

### 3.1 데이터 소스

```sql
FROM MCP_CANCEL_REQUEST@DL_MCP A   -- M포탈 해지신청 테이블 (DB Link)
   , MSP_JUO_SUB_INFO B            -- MSP 가입자 정보
   , MSP_REQUEST_DTL C             -- MSP 신청 상세 (요금제 등)
WHERE A.CONTRACT_NUM = B.CONTRACT_NUM
  AND B.CONTRACT_NUM = C.CONTRACT_NUM  -- ⚠️ INNER JOIN → MSF 건 누락 위험
```

> **⚠️ TOBE 이슈**: `MSP_REQUEST_DTL C` INNER JOIN 으로 MSF 신규 해지 건(C에 데이터 없는 경우) 목록 미표시 가능  
> → Z10 설계서에서 `LEFT OUTER JOIN`으로 변경하도록 명시

### 3.2 주요 조회 컬럼

| 컬럼 | 설명 |
|------|------|
| `A.CUST_REQ_SEQ` | 해지신청 시퀀스 (처리 키) |
| `A.CONTRACT_NUM` | 계약번호 |
| `B.SUB_LINK_NAME` | 고객명 |
| `CRYPTO_AES256.DEC_AES(B.USER_SSN)` | 주민번호 복호화 (생년월일/성별 파싱) |
| `A.CANCEL_MOBILE_NO` | 해지 신청 번호 |
| `A.RECEIVE_MOBILE_NO` | 연락가능 연락처 |
| `C.LST_RATE_CD / LST_RATE_NM` | 현재 요금제 |
| `B.MODEL_NAME / B.INTM_SRL_NO` | 단말모델 / 일련번호 |
| `A.PROC_CD` | 처리상태 코드 (RC/CP/BK 등) |
| `A.SCAN_ID` | **NEW_SCAN_ID** (MSF SCAN 전송 결과) |
| `B.SCAN_ID` | **OLD_SCAN_ID** (MSP 기존 서식지) |
| `A.SURVEY1_TEXT / SURVEY2_TEXT` | 해지이유 설문 |
| `A.SURVEY_SCORE / SURVEY_SUGGESTION` | 만족도 / 개선의견 |

| 검색구분 | 파라미터 | SQL 조건 |
|---------|---------|---------|

## 5. 처리 확정 (`updateCanCsl`) — 핵심 로직

`canCustService.updateCanCsl(searchVO, loginInfo.getUserId())` 내부에서 아래 3단계 처리:

### 5.1 SCAN 합본 처리 (PROC_CD = 'CP' 일 때)

MSF가 전송한 신청서(`NEW_SCAN_ID = A.SCAN_ID`)를 기존 MSP 서식지(`OLD_SCAN_ID = B.SCAN_ID`)에 합본.

**Step 1** — 파일 번호 조회
```sql
SELECT MAX(FILE_NUMBER) + 1 AS fileNum
  FROM EMV_SCAN_FILE@DL_MCP
 WHERE SCAN_ID = #{oldScanId}
```

**Step 2** — 신규 파일을 기존 SCAN에 합본 INSERT
```sql
INSERT INTO EMV_SCAN_FILE@DL_MCP
  (FILE_ID, SCAN_ID, FILE_NUMBER, FILE_SIZE, ...)
SELECT FILE_ID, #{oldScanId}, #{fileNum} + FILE_NUMBER, ...
  FROM EMV_SCAN_FILE@DL_MCP
 WHERE SCAN_ID = #{newScanId}
```

**Step 3** — SCAN 마스터 파일 수 갱신
```sql
UPDATE EMV_SCAN_MST@DL_MCP
   SET SCAN_FILE_CNT = #{maxFileCnt}
 WHERE SCAN_ID = #{oldScanId}
```

### 5.2 PROC_CD 갱신

```sql
UPDATE MCP_CANCEL_REQUEST@DL_MCP
   SET MEMO       = #{memo}
     , PROC_CD    = #{procCd}
     , RVISN_ID   = #{sessionUserId}
     , RVISN_DTTM = SYSDATE
 WHERE CUST_REQ_SEQ = #{custReqSeq}
   AND PROC_CD != 'CP'   -- 이중 처리 방지
```

## 9. PROC_CD UPDATE 이후 실제 해지처리 흐름 분석

### 9.1 핵심 발견 사항

> **⚠️ 중요**: MSP `updateCanCsl`은 **DB 상태값만 변경**한다. 실제 KT 실시스템(M플랫폼) 해지는 별도 **일괄직권해지 배치(BATCH00233)** 가 처리한다.

### 9.2 전체 실해지 처리 흐름

```
[MSP 담당자]
    │
    ▼
 updateCanCsl() — CanCustService.java:205
    │
    ├─ ① CP 처리 시 SCAN 합본
    │       getCanCslFileNum()       → EMV_SCAN_FILE@DL_MCP MAX(FILE_NUMBER)+1 조회
    │       insertCanCslEmvFile()    → NEW_SCAN_ID 파일을 OLD_SCAN_ID에 INSERT 합본
    │       updateCanCslEmvFileMst() → EMV_SCAN_MST@DL_MCP.SCAN_FILE_CNT 갱신
    │
    └─ ② PROC_CD 상태값만 UPDATE
            UPDATE MCP_CANCEL_REQUEST@DL_MCP
               SET PROC_CD = #{procCd}  -- RC/RQ/CP/BK
            WHERE PROC_CD != 'CP'        -- 이중처리 방지

            ※ 여기까지가 MSP updateCanCsl의 범위
            ※ 실제 KT 실시스템 해지는 아래 배치로 처리

[일괄직권해지 배치 — BATCH00233]
    │
    ├─ 트리거: MSP 담당자가 '일괄직권해지 화면'에서
    │          엑셀 업로드 → MSP_CAN_REQ 등록 → 배치 요청(retryCanBatchRequest)
    │
    ├─ 대상: MSP_CAN_REQ WHERE REQ_STATUS='N' (요청대기)
    │
    ├─ 처리: M플랫폼(KT 실시스템) 해지 API 호출
    │       → TCP_PRGR_STAT_CD = 'EP4', TCP_RSLT_CD = '0000' 이면 해지완료
    │       → MSP_CAN_REQ.REQ_STATUS = 'Y' (요청완료)
    │
    └─ 결과 반영
            MSP_JUO_SUB_INFO.SUB_STATUS = 'C' (해지)
            MSP_CAN_REQ.TCP_RSLT_DTTM   = 처리일시

[msc-batch — CancelConsultController]
    └─ @Scheduled(cron="0 30 2 1 * *") — 매월 1일 02:30
         해지상담 신청 중 6개월 경과 미처리 건(PROC_CD != 'CP') DELETE
         → MCP_CANCEL_REQUEST에서 오래된 미처리 건 정리
```

### 9.3 `updateCanCsl` 상세 코드 분석

```java
// CanCustService.java:205~236
@Transactional(rollbackFor=Exception.class)
public int updateCanCsl(CanCustVO vo, String usrId) throws EgovBizException {

    // ① CP(처리완료) 일 때만 SCAN 합본
    if("CP".equals(vo.getProcCd())) {
        String maxFileNum = canCustMapper.getCanCslFileNum(vo.getOldScanId());
        vo.setFileNum(maxFileNum);
        resultCnt = canCustMapper.insertCanCslEmvFile(vo);   // 파일 합본
        int newFileCnt = Integer.parseInt(maxFileNum) + resultCnt;
        canCustMapper.updateCanCslEmvFileMst(vo);            // 파일수 갱신
    }

    // ② 상태값 UPDATE (DB만 변경, 실시스템 해지 X)
    resultCnt = canCustMapper.updateCanCslProcCd(vo);
    return resultCnt;
}
```

> **주석 확인**: 코드 내 주석 `NMCP_CUST_REQUEST_CALL_LIST 테이블에 메모와 상태를 업데이트`는  
> 실제 실행 SQL은 `MCP_CANCEL_REQUEST@DL_MCP` UPDATE (주석 내용과 불일치)

### 9.4 일괄직권해지(BATCH00233)와 해지상담신청의 관계

| 구분 | 해지상담신청 (canCanCslMgmt) | 일괄직권해지 (canBatchMgmt) |
|------|---------------------------|--------------------------|
| 대상 | MCP 고객 온라인 신청 건 | 담당자가 대상 선별하여 엑셀 업로드 |
| 저장 테이블 | `MCP_CANCEL_REQUEST@DL_MCP` | `MSP_CAN_REQ` |
| 실해지 처리 | **없음** (상담 후 수동 처리) | **BATCH00233이 M플랫폼 직접 호출** |
| PROC_CD | RC → CP (DB 상태만) | REQ_STATUS: N → Y (실해지 완료) |
| 사용권한 | 전체 (본사 엑셀만 제한) | **본사(orgTypeCd=10)만 가능** |

### 9.5 해지상담신청 PROC_CD 이후 실제 해지 처리 경로

```
[결론]
MSP updateCanCsl PROC_CD='CP' 저장
    → 실시스템 해지 없음
    → KT 상담원이 별도 채널(전화)로 M플랫폼 직접 처리 또는
    → 일괄직권해지 배치(BATCH00233)로 별도 등록하여 처리

즉, '해지상담신청' 화면의 PROC_CD='CP'는
'고객과 통화 후 해지 진행 확인' 상태이며,
실제 KT 번호 해지는 M플랫폼을 통해 별도 수행된다.
```

### 9.7 EP0 — M플랫폼 실시간 해지 API

**EP0**는 MCP 포탈이 M플랫폼(KT 실시스템)에 직접 호출하는 **실시간 해지 처리 API**이다.  
해지상담신청(canCanCslMgmt) 화면의 PROC_CD 갱신과는 **별개 채널**로 동작한다.

#### 9.7.1 EP0 개요

| 항목 | 내용 |
|------|------|
| **API 코드** | `EP0` |
| **서비스명** | `OsstCanSvcPrcService` |
| **Operation** | `osstCanPrc` (일반) / `osstCanM2mPrc` (M2M) |
| **연동 방식** | MVNO → M플랫폼 (Socket/REST 양방향) |
| **처리 결과** | M플랫폼 → MVNO로 Socket 방식으로 결과 통보 |
| **인터페이스 설계서** | `C1.MMSP-DS-06 IF_0033(일반), IF_0034(M2M)` |
| **구현 위치** | `MPlatformMapper.xml` `getXmlMessageEP0` |

#### 9.7.3 EP0 응답 및 상태코드


### 9.8 EP0 vs BATCH00233 vs updateCanCsl 비교

| 구분 | `updateCanCsl` (MSP) | `EP0` (MCP 온라인) | `BATCH00233` (MSP 일괄) |
|------|---------------------|-------------------|------------------------|
| **트리거** | 담당자 저장버튼 | MCP 고객 온라인 신청 완료 | 담당자 엑셀 업로드 후 배치 실행 |
| **대상 테이블** | `MCP_CANCEL_REQUEST@DL_MCP` | `MCP_REQUEST_OSST` | `MSP_CAN_REQ` |
| **실시스템 해지** | ❌ DB 상태만 변경 | ✅ M플랫폼 실시간 호출 | ✅ M플랫폼 배치 호출 |
| **처리 상태코드** | `PROC_CD`: RC→CP | `PRGR_STAT_CD`: EP0→EP4 | `REQ_STATUS`: N→Y |
| **결과 통보** | 없음 | Socket 비동기 통보 | DB 폴링 |
| **사용 채널** | MSP 담당자 화면 | MCP 포탈 (고객 직접) | MSP 담당자 화면 |
