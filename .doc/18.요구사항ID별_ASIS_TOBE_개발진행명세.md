### 0.2 설계 부족 사항 (Gap Analysis)

#### [GAP-01] DB 테이블 이원화 — 최우선 결정 필요

**현황**: 명의변경·해지 신청서가 레거시 NMCP 테이블에 저장되고 있음.

| 도메인 | 현재 저장 테이블 | 신규 설계 테이블 (DS-05) |
|--------|----------------|----------------------|
| 명의변경 양도인 | `NMCP_NFL_CHG_TRNS` | `MSF_REQUEST_NAME_CHG` + `MSF_REQUEST_NAME_TRNS` |
| 명의변경 양수인 | `NMCP_NFL_CHG_TRNSFE` | `MSF_REQUEST_NAME_CHG` (통합) |
| 서비스해지 | `NMCP_NFL_CNCL_APY` | `MSF_REQUEST_CANCEL` |
| 서비스변경 | (미구현) | `MSF_REQUEST_SVC_CHG` + `MSF_REQUEST_SVC_CHG_DTL` |

**문제**: 신규 서비스 테이블(`MSF_REQUEST_*`)은 DS-05에서 확정됐으나, 현재 구현은 레거시 테이블에 저장 중.
공통 신청서 테이블(`MSF_REQUEST`, `MSF_REQUEST_CSTMR`, `MSF_REQUEST_CLAUSE` 등)도 도메인과 연계 미결정.

**결정 필요**: 레거시 NMCP 테이블 유지 or MSF 신규 테이블로 전환. 전환 시 Mapper·DTO·XML 전면 수정 필요.

---

#### [GAP-02] 신청서 공통 저장·전송 플로우 미설계

**현황**: apply() 호출 시 DB 저장만 하고 후속 전송 처리가 없음.

**설계 필요 내용**:

```
[apply() 완료]
    ↓
① MSF 신청서 테이블 저장 (MSF_REQUEST_* 계열)
    ↓
② eFormSign 녹취·서명 연동 (MSF_REQUEST_REC)
    ↓
③ M전산 전송 (mcp-api)
    ↓
④ M플랫폼 전송 (MC0/MP0 등)
    ↓
⑤ 이미징 (IF_0035 PDF 생성, IF_0038 K-NOTE 전송)
    ↓
⑥ SMS/카카오 알림 (IF_0031/IF_0030)
```

- 각 단계 실패 시 롤백 전략 또는 재처리 방안 미정
- 어느 단계에서 사용자 완료 응답을 보낼지 미결정
- `FormSendController` / `FormSendSvc` 공통 모듈 설계 필요 (20.11절)

---

#### [GAP-03] 인증·세션·권한 구조 미설계

**현황**: 모든 API가 인증 없이 호출 가능. 대리점 코드·사용자 ID가 요청 바디에 직접 포함됨.

**설계 필요 내용**:
- 로그인 방식: JWT 토큰 or Spring Session 기반 선택
- 대리점 코드(agencyCode) 검증: 로그인 세션에서 추출 vs 요청 바디 수신 방식 결정
- Spring Security 필터 체인 구성 여부
- 대리점/판매점 정보 조회 API (C09, C75) 설계 → `AgentInfoController` (20.2절)

---

#### [GAP-04] 서명(eFormSign) 통합 플로우 미설계

**현황**: `formOwnChg/FormOwnChgStep3.vue`, `formSvcCncl/FormSvcCnclStep3.vue` 화면에 서명 UI가 구현되어 있으나, eFormSign 연동 스펙 미정의.

**설계 필요 내용**:
- eFormSign SDK 초기화·서명 캡처 → 서명 데이터를 백엔드로 전달하는 API 경로
- 서명 완료 검증 로직 (`MSF_REQUEST_REC` 테이블 저장)
- 녹취 연동 여부 및 파일 저장 방식 (IF_0035/IF_0038)

---

#### [GAP-05] 공통 예외 처리 및 에러 응답 포맷 미표준화

**현황**: 각 Controller·Service가 개별 try-catch로 에러 처리. 에러 응답 형태가 도메인마다 다름.

| 도메인 | 현재 에러 응답 형태 |
|--------|-----------------|
| 서비스해지 | `CancelApplyVO.fail(message)` |
| 명의변경 | `OwnChgApplyVO` (success/message) |
| 부가서비스 | `Map<String,Object>` (success/message) |

**설계 필요 내용**:
- 공통 응답 DTO (`MsfApiResponse<T>`) 정의: `success`, `code`, `message`, `data`
- `GlobalExceptionHandler` (@RestControllerAdvice) — M플랫폼 오류코드 → 사용자 메시지 매핑
- M플랫폼 연동 오류 시 에러 코드 분류 기준 (`resultCode` 처리 규칙)

---

#### [GAP-06] M플랫폼 연동 미완성 (Stub 상태)

| 연동 코드 | 기능 | 현재 상태 | 해결 방안 |
|----------|------|----------|----------|
| MC0 | 명의변경·해지 가능 여부 사전체크 | `eligible()` → 항상 true 반환 | MplatFormSvc.mc0PreCheck() 구현 필요 |
| X18 | 잔여요금·위약금·분납금 조회 | `getRemainCharge()` → null 반환 | MplatFormSvc.getRemainCharge() 구현 필요 |
| MP0 | 명의변경 후처리 (명의변경 실행) | 미구현 | OwnChgApplySvcImpl에 MP0 연동 추가 |

---

#### [GAP-07] 신청서 조회·미리보기·이력 미설계

**현황**: apply() 후 신청서 번호("IDN-xxx", "CNC-xxx")는 반환되지만, 해당 번호로 신청서를 다시 조회하거나 미리보기하는 API 없음.

**설계 필요 내용**:
- 도메인별 신청서 단건 조회: `GET /api/v1/ident/apply/{applicationNo}`, `GET /api/v1/cancel/apply/{applicationNo}`
- 신청서 목록 조회 (대리점 기준): 페이징·검색 조건
- 리포트 이미징 연동 (IF_0035) — PDF 생성 경로·파일 관리 방식
- 이력 테이블 설계 (상태 변경 추적)

### 21.2 화면·UI (신규 설계 적용)

- **스텝 컴포넌트 구조 (MsfStepView 기반)**
  - 공통 스텝 프레임: `MsfStepView.vue` (`/step/:domain` 라우트). 도메인별 스텝 배열은 `src/stores/msf_step.js`에 정의.
  - 도메인별 컴포넌트 경로:
    - 서비스변경: `formSvcChg/FormSvcChgStep1/2/3.vue`
    - 명의변경: `formOwnChg/FormOwnChgStep1/2/3.vue`
    - 서비스해지: `formSvcCncl/FormSvcCnclStep1/2/3.vue`
  - 각 Step 컴포넌트 인터페이스: `defineProps({ isActive: Boolean })`, `emit('complete', boolean)`, `defineExpose({ save })` (save() → boolean 반환)
  - 동적 임포트: `comp.utils.js:getStepComponent(dir, file)` → `import('@/components/${dir}/${file}.vue')`
  - msf-web GitHub 저장소: `https://github.com/turtler501m-ai/msf-web`
- **신청서 화면 구조**
  - **고객 → 상품 → 동의** 3단계. 서브메뉴/서브타이틀 밑에 밑줄, **왼쪽 라벨 + 오른쪽 입력박스** 그리드. 라벨 칸 너비는 모든 화면에서 동일 유지.
- **화면 설계서 기준**
  - `uidoc` 문서의 **위치·박스·입력 항목**에 맞춰 배치. 2026.03.06 화면설계서대로 입력 항목 맞춤 적용.
- **서비스해지**
  - `uidoc/서비스해지 최신 20260310` 반드시 참고. M포탈과 다른 레이아웃(가입유형 통합 등) 적용.
- **참고 화면 자료**
  - 30번 문서의 참고자료 컬럼(예: 서비스변경_01 메인화면.png, 서비스변경_02 1단계 고객정보입력.png, 서비스변경_03 부가서비스 선택팝업.png 등)으로 화면 검증.

### 21.3 기술 스택·구조 (AS-IS와 다른 부분)

- **프론트**  
  - Vue3, JavaScript, Vite, Pinia, Vue Router, TailwindCSS. **TypeScript 사용 금지.**  
  - AS-IS JSP/JS 구조를 **복사하지 않고** 동일 기능을 Vue 컴포넌트로 신규 구현.
- **백엔드**  
  - Java, Spring Boot, MyBatis. 루트 패키지 `com.ktmmobile.msf`, 도메인별 `formSvcChg`, `formOwnChg`, `formSvcCncl`, `formComm`.
- **API**  
  - M플랫폼 연동 방식·스펙은 MCP와 동일. 신규 REST API 경로·Req/Res DTO는 본 문서 1~20절 및 17번 문서 구조 따름.
