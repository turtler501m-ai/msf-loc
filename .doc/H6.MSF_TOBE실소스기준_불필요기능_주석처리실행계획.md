## 1. 문서 목적

본 문서는 실제 TOBE 복사 소스 기준으로, `servicechange` 하위 Controller/Service 중 ASIS 복사 잔재로 판단되는 기능을 주석처리 관점에서 정리한 실행 계획서다.

핵심 원칙은 다음과 같다.

1. H1에서 사용 중인 기능은 화면 메서드라도 주석처리 금지다.
2. H1에서 `🚫 제외`, H2/H4/H5에서 `ASIS 전용`, `범위 외`, `팝업`, `탭목록`, `KT 결합 제외`로 확정된 기능은 주석처리 우선 대상이다.
3. 이미 주석 처리되어 있는 메서드도 “정리 완료”로 보지 않는다.
4. 즉, 파일 안에 남아 있는 주석 메서드는 모두 잠재적 ASIS 복사 잔재로 보고 별도 정리 대상으로 관리한다.
5. `prodSendScan` 계열은 불필요 기능이 아니라 “ASIS 복사 구현 재사용 금지” 대상으로 본다.

---

## 1-1. 주석처리 작업 규칙

실제 코드 작업 시에는 아래 규칙을 공통 적용한다.

1. ASIS 복사 원문은 삭제하지 않는다.
   - 메서드, 분기, import, 필드, 시그니처를 제거하지 말고 주석으로 남긴다.
2. TOBE에서 제외하는 이유를 바로 위에 짧게 남긴다.
   - 기능명과 제외 사유만 한 줄로 이해되게 작성한다.
3. 새로 추가하는 설명 주석은 `TOBESKIP:` 접두어를 사용한다.
   - 예: `// TOBESKIP: 가입신청서 화면은 사용하지 않아 URL 매핑만 막고 원본 로직은 보존한다.`
4. URL 차단이 목적이면 `@RequestMapping`만 우선 주석처리한다.
   - 메서드 본문은 그대로 남겨 원문 추적이 가능해야 한다.
5. 처리 분기 제외가 목적이면 분기 블록 전체를 주석으로 감싼다.
   - 남겨야 하는 TOBE 기능 분기는 그대로 유지한다.
6. 사용하지 않게 된 import, 필드, 서비스 시그니처도 삭제 대신 주석 보존을 우선한다.
7. 주석처리 후에는 반드시 빌드로 문법, import, 인터페이스 불일치를 확인한다.

이 규칙은 이후 Wave 2~Wave 5 작업에도 동일하게 적용한다.

## 10. 우선순위 계획

### 10-1. 우선순위 기준

우선순위는 아래 순서로 판단한다.

1. 현재 H1에서 `🚫 제외`가 명확한가
2. 프론트/API 호출 영향이 거의 없는가
3. 메서드 단독 주석처리가 가능한가
4. Controller와 Service가 섞여 있지 않고, 부작용 범위가 좁은가
5. 운영/배치/외부연동 영향이 없는가

이 기준으로 웨이브를 나누면 다음과 같다.

### 10-2. Wave 1: 저위험 즉시 주석처리

목표

- 가장 명확한 불필요 기능부터 제거해 코드 가독성을 빠르게 높인다.
- 기동 영향이 거의 없는 메서드만 먼저 정리한다.

대상

- `MsfFarPricePlanController.doRegServicePop`
- `MsfFarPricePlanController.farPriceCtgTab1Ajax`
- `MsfFarPricePlanController.getFarPricePlanListAjax2`
- `MsfFarPricePlanController.getFarPricePlanListAjax3`
- `MsfFarPricePlanController.farPriceContent`
- `MsfPauseController.suspenPosHisPop`
- `MsfMyShareDataController.domyCntrListAjax`
- `MsfCustRequestController.reqJoinForm`
- `MsfCustRequestController.reqUsimPuk`
- `MsfCustRequestService.insertCustRequestCallList`
- `MsfCustRequestService.insertCustRequestJoinForm`
- `MsfCustRequestService.getUsimPukByMP`
- `MsfCustRequestServiceImpl.insertCustRequestCallList`
- `MsfCustRequestServiceImpl.insertCustRequestJoinForm`
- `MsfCustRequestServiceImpl.getUsimPukByMP`

작업 방식

- 메서드 선언과 매핑을 주석처리
- 동일 이름의 인터페이스/구현체를 같이 맞춘다
- 사용하지 않게 된 import를 같이 정리한다

검증

- 컴파일 에러 없음
- URL 매핑 충돌/누락 없음
- H1/H4/H5/H6 문서와 불일치 없음

### 10-3. Wave 2: KT 결합 계열 일괄 주석처리

목표

- `아무나SOLO`와 무관한 KT 결합 잔재를 묶음 정리한다.

대상

- `MsfCombineController.combineWireless`
- `MsfCombineController.combineWirelessComplete`
- `MsfCombineController.checkCombine`
- `MsfCombineController.childCertify`
- `MsfCombineController.childCertifyKt`
- `MsfCombineController.childCertifyMore`
- `MsfCombineController.childVerifyAjax`
- `MsfCombineController.regCombine`
- `MsfCombineController.regCombineKt`
- `MsfCombineController.eSimWatchInfo`
- `MsfCombineController.getCombiSvcList`
- `MsfCombineController.combineSoloType`
- 연관 서비스:
  - `MsfMyCombinationSvc`
  - `MsfMyCombinationSvclmpl`
  - 단, `아무나SOLO`에 필요한 메서드는 제외하고 KT 결합 계열만 식별 후 주석처리

작업 방식

- Controller부터 막지 말고, 먼저 KT 결합 관련 메서드 목록을 서비스까지 매핑한다
- `아무나SOLO`와 공유하는 DTO/Helper는 건드리지 않는다
- KT 결합 전용 메서드만 주석처리한다

검증

- `combineSelf`, `checkCombineSelf`, `regCombineSelf` 정상 유지
- `MsfChildInfoService`가 여전히 필요한지 재평가
- KT 결합 URL이 더 이상 외부에 노출되지 않는지 확인

### 10-4. Wave 3: 주석 상태 잔재 정리

목표

- 이미 주석되어 있지만 파일에 남아 있는 ASIS 복사 잔재를 구조적으로 정리한다.

대상

- `MsfPayInfoService`
- `MsfPayInfoServiceImpl`
- `MsfPaywayService`
- `MsfPaywayServiceImpl`
- `MsfRegSvcServiceImpl`의 과거 시그니처 주석 블록

작업 방식

- 먼저 “지금 문서상 유지해야 하는 기능과 무관한지”를 확인한다
- 무관하다면 주석 블록을 더 남겨둘지, 완전 제거 후보로 표시할지 결정한다
- 한 번에 삭제가 부담되면 `LEGACY-ASIS` 구획 주석으로 모아둔다

검증

- 활성 코드와 주석 잔재가 혼동되지 않는 상태인지 확인
- 관련 DTO/Mapper/유틸이 고아 상태가 되는지 확인

### 10-5. Wave 4: 운영성/보류 항목 재판단

목표

- 1~3차 정리 후에도 남는 회색지대 항목을 결정한다.

대상

- `MsfFarPricePlanController.resChangeAddPrd`
- `MsfCustRequestController.reSpnsrPlcyDc`
- `MsfCustRequestController.getUserInfoAjax`

작업 방식

- 호출처 확인
- 운영 의존성 확인
- 필요 시 “주석처리” 대신 “보류 표기”로 유지

검증

- 운영 배치/스크립트 영향 여부 확인
- 보험 기능 영향 여부 확인

### 10-6. Wave 5: 재사용 금지 항목 분리

목표

- 주석처리와 재구현 대상을 혼동하지 않도록 별도 트랙으로 관리한다.

대상

- `MsfCustRequestScanServiceImpl.prodSendScan`
- `MsfCustRequestScanServiceImpl.createXml`
- `MsfCustRequestScanServiceImpl.saveXML`
- `MsfCustRequestScanServiceImpl.xmlFileSend`

작업 방식

- 직접 주석처리보다 신규 구현 태스크로 분리
- 실제 신규 구현 전까지는 기존 호출 영향도를 파악만 한다

검증

- 포시에스 전환 계획과 연결되었는지 확인
- “불필요 기능” 문맥으로 잘못 삭제되지 않도록 문서화

### 10-7. 추천 작업 순서 요약

1. Wave 1로 저위험 항목부터 정리
2. Wave 2로 KT 결합 계열 일괄 정리

4. Wave 4로 보류 항목 재판단
5. Wave 5는 별도 재구현 트랙으로 분리

## 11. 체크리스트

### 11-1. 주석처리 전

- H1에서 사용 중 항목이 아닌지 확인
- 프론트 호출 URL이 남아 있지 않은지 확인
- Swagger, 테스트, 배치 호출이 없는지 확인
- 같은 이름의 Service/Mapper 메서드가 다른 곳에서 재사용되지 않는지 확인
- 이미 주석된 메서드라도 DTO, Mapper, 유틸이 그 잔재 때문에 남아 있지 않은지 확인

### 11-2. 주석처리 후

- 기동 시 중복/누락 매핑 오류가 없는지 확인
- 사용하지 않는 import 제거
- 인터페이스/구현체 시그니처 불일치가 없는지 확인
- 주석 상태 잔재 블록이 문서화한 범위만 남았는지 확인
- H4, H5, H6 문서와 실제 코드 상태가 일치하는지 확인
