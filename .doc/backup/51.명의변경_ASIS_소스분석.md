# 명의변경 AS-IS 소스 분석

**분석 대상**: MCP 포털·관리자 명의변경 관련 소스  
**관련 문서**: `ktm/doc/명의변경신청서_UI설계서_분석.md` (TO-BE UI 설계서)

---

## 1. 개요

AS-IS에서 명의변경은 **두 가지 경로**로 존재한다.

| 구분 | 설명 | 비고 |
|------|------|------|
| **1) 명의변경 전용 플로우** | 양도인·양수인 신청/조회/상태관리. 테이블 `NMCP_NFL_CHG_TRNS`, `NMCP_NFL_CHG_TRNSFE`. 포털 CsTransfer 서비스 + 관리자 리스트/상세/처리. | 별도 URL/컨트롤러는 코드베이스에서 미확인(매퍼 namespace 불일치 가능) |
| **2) 가입신청서(앱폼) 내 명의변경** | `MCP_REQUEST_CHANGE` 테이블. 신규/변경 가입 시 항상 INSERT(현재는 빈 값). 스캔/리포트 시 `NAME_CHANGE_*` 사용. | 명의변경 전용 UI 없이, 앱폼 공통 저장·스캔에서만 사용 |

---

## 2. 명의변경 전용 플로우 (양도인/양수인)

### 2.1 테이블

| 테이블 | 용도 |
|--------|------|
| **NMCP_NFL_CHG_TRNS** | 양도인(현재 명의자) 신청 정보. 신청번호(TRNS_APY_NO), 비밀번호(TRNS_PASS), 양도인/양수인 이름·연락처·본인확인방법·주민/면허 정보, 상태(TRNS_STATUS_VAL) 등. |
| **NMCP_NFL_CHG_TRNSFE** | 양수인(명의변경 후 고객) 정보. TRNSFE_APY_NO로 양도인 신청과 연결. 이름, 전화번호, 주민번호, 본인확인방법, 주민증/면허 발급정보 등. |

**상태값(TRNS_STATUS_VAL)**  
- 1: 대기중  
- 2: 신청완료  
- 3: 처리완료  
- 4: 부재중(기한초과 시 자동 변경)  
- 5: 기한초과  

**시퀀스**  
- `SQ_TRNS_SEQ`, `SQ_TRNS_APY_SEQ`, `SQ_TRNSFE_SEQ` (Oracle)

---

### 2.2 포털 – 서비스·DAO

**패키지**: `com.ktmmobile.mcp.cs`

| 계층 | 클래스 | 역할 |
|------|--------|------|
| Service | `CsTransferSvc`, `CsTransferSvcImpl` | 명의변경 비즈니스 로직 |
| DAO | `CsTransferDao`, `CsTransferDaoImpl` | 양도인/양수인 CRUD (namespace `CsTransfer_Query`) |
| DAO | `TransferDao`, `TransferDaoImpl` | 리스트/상세/기한초과 처리 (동일 namespace) |

**CsTransferSvc 메서드**

| 메서드 | 설명 |
|--------|------|
| `telNoList(String userId)` | 기한초과 처리(`trnsStausEdit`) 후 해당 사용자 회선 목록(CNTR_MOBILE_NO) 조회. |
| `grantorRegSave(NflChgTrnsDto)` | 양도인 신청 저장. NMCP_NFL_CHG_TRNS INSERT, TRNS_STATUS_VAL='1'. |
| `assigneeReg(UserSessionDto)` | 로그인 사용자 이름(`userSession.name`)으로 양도인이 입력한 양수인 이름(TRNS_TRNSFE_NM)과 일치하는 양도인 신청 1건 조회. |
| `checkTrnsApyNoAjax(NflChgTrnsfeDto)` | 신청번호(tmpTrnsApyNo)+비밀번호(tmpTrnsPass)로 양도인 신청 조회. |
| `assigneeRegSave(NflChgTrnsfeDto)` | 양수인 정보 NMCP_NFL_CHG_TRNSFE INSERT 후 TRNS_STATUS_VAL='2'(신청완료)로 업데이트. |
| `checkDupl(NflChgTrnsfeDto)` | 동일 TRNSFE_APY_NO에 대한 양수인 등록 건수 조회(중복 방지). |

**주의**: 포털 DAO는 MyBatis namespace **`CsTransfer_Query`** 를 사용하나, 리포지토리에는 해당 namespace의 XML이 없음. 관리자(mcp-admin-was)에는 **`CsTransferMapper`** namespace로 동일 ID의 쿼리가 있음. 배포 시 매퍼 공유/별도 XML 존재 여부 확인 필요.

### 2.3 DTO

| DTO | 용도 |
|-----|------|
| **NflChgTrnsDto** | 양도인: trnsApyNo, trnsNm, trnsMobileNo, trnsPass, trnsMyslfConfMeth, 주민/면허 발급정보, trnsTrnsfeNm, trnsTrnsfeMobileNo, trnsStatusVal, authNum 등. |
| **NflChgTrnsfeDto** | 양수인: trnsfeApyNo, trnsfeNm, trnsfeMobileNo, trnsfeResno, trnsfeMyslfConfMeth, 주민/면허 발급정보. 임시: tmpTrnsApyNo, tmpTrnsPass, tmpStatusVal 등. |
| **TransferDto** | 관리자 리스트/상세: 양도·양수인 필드 + BOARD_NUM, treatVal, searchValue, divVal 등 검색/페이징. |
| **CsMcpUserCntrDto** | 회선 목록 조회: CNTR_MOBILE_NO. |

### 2.4 관리자 – 명의변경

**매퍼**: `mcp-admin-was/src/main/resources/sql/map/CsTransferMapper.xml` (namespace `CsTransferMapper`)

| ID | 설명 |
|----|------|
| `trnsStausEdit` | 7일 초과·상태 1인 건을 상태 4(부재중)로 일괄 업데이트. |
| `countTransferList`, `transferList` | 검색(접수번호/작성자/신청일/처리일, treatVal) 조건으로 목록 카운트/조회. |
| `transferDetail` | TRNS_APY_NO로 양도인·양수인 조인 상세 조회. |
| `transferDetailSave` | TRNS_STATUS_VAL, CONFIRM_MEMO, AMD_ID, AMD_DT 업데이트. |
| `transferDetailDel` | 개인정보 마스킹(삭제) 처리 후 TRNS_STATUS_VAL='3' 등으로 업데이트. |

관리자 쪽 Controller/URL은 본 분석에서 미추적. 위 매퍼를 사용하는 Admin 컨트롤러/서비스가 별도 존재할 수 있음.

## 3. 가입신청서(앱폼) 내 명의변경

### 3.1 테이블 MCP_REQUEST_CHANGE

| 컬럼 | 설명 |
|------|------|
| NAME_CHANGE_NM | 명의변경_고객명 |
| NAME_CHANGE_TEL_FN/MN/RN | 명의변경_전화번호 (앞/중간/끝) |
| NAME_CHANGE_RRN | 명의변경_주민등록번호 |
| NAME_CHANGE_PINSTALLMENT | 완납=P, 승계=C |
| REQUEST_KEY | 가입신청 키 (MCP_REQUEST 등과 연관) |
| SYS_RDATE | 등록일시 |

---

### 3.2 AppformSvcImpl

- **신규/변경 가입 저장 시**  
  - `insertMcpRequestChange(appformReqDto)` 호출.  
  - **AppFormMapper.xml** `insertMcpRequestChange`: NAME_CHANGE_* 컬럼은 모두 **빈 문자열('')** 로 INSERT. 즉, 현재 앱폼에서는 명의변경 전용 입력값을 채우지 않음.
- **수정 저장 시**  
  - `deleteMcpRequestChange` 후 `insertMcpRequestChange` 재호출. 동일하게 requestKey 기준으로 명의변경 행은 빈 값으로 유지.

---

### 3.3 AppformDao / AppFormMapper.xml

- `insertMcpRequestChange`: REQUEST_KEY, SYS_RDATE와 NAME_CHANGE_* (빈 값), NAME_CHANGE_PINSTALLMENT (빈 값) INSERT.  
- `deleteMcpRequestChange`: REQUEST_KEY로 MCP_REQUEST_CHANGE DELETE.

---

### 3.4 스캔/리포트 (ScanSvcImpl)

- **NAME_CHANGE_RRN** 이 있으면 복호화 후 스캔 요청 맵에 포함.  
- 주민번호 7번째 자리로 성별 구분해 `NAME_CHANGE_RRN_M`, `NAME_CHANGE_RRN_W` 등 플래그 세팅.  
→ 명의변경 신청서 등에서 명의변경 주민번호가 채워진 경우, 스캔/PDF 생성 시 반영되는 구조이나, 현재 INSERT는 빈 값이므로 실제 데이터는 “명의변경 전용 플로우”나 향후 TO-BE에서 넣는 값에 의존.

---

### 3.5 McpRequestChangeDto / AppformReqDto

- **McpRequestChangeDto**: nameChangeNm, nameChangeTelFn/Mn/Rn, nameChangeRrn, nameChangePinstallment.  
- **AppformReqDto**: 명의변경 필드가 있으면 BeanUtils로 McpRequestChangeDto에 복사 후 사용. 현재 insertMcpRequestChange는 AppformReqDto를 넘기며, XML에서 고정 빈 값으로 넣고 있음.

## 4. 데이터 흐름 요약

### 4.1 명의변경 전용 (양도인 → 양수인)

1. **양도인**: 휴대폰·본인확인·비밀번호·양수인 이름·연락처 입력 후 `grantorRegSave` → NMCP_NFL_CHG_TRNS INSERT (상태 1).  
2. **양수인**:  
   - 로그인 후 `assigneeReg`로 “내 이름”으로 등록된 신청 조회.  
   - 또는 신청번호+비밀번호로 `checkTrnsApyNoAjax` 조회.  
3. 양수인 정보 입력 후 `assigneeRegSave` → NMCP_NFL_CHG_TRNSFE INSERT, NMCP_NFL_CHG_TRNS 상태 2(신청완료)로 변경.  
4. **관리자**: 리스트/상세 조회, 기한초과 자동 처리(7일 초과 시 4), 처리완료/메모/개인정보 삭제 처리.

### 4.2 가입신청서(앱폼)와의 관계

- 앱폼 제출 시 MCP_REQUEST_CHANGE에는 항상 한 행이 INSERT되나 **명의변경 고객명/전화/주민/완납승계는 비어 있음**.  
- 스캔/리포트는 NAME_CHANGE_* 가 채워져 있으면 활용하는 구조만 존재.  
- 따라서 **현재 AS-IS에서는 “명의변경 전용 플로우”와 “앱폼 MCP_REQUEST_CHANGE”가 서로 다른 목적**으로 사용됨.  
  - 전자: 양도인/양수인 신청·관리.  
  - 후자: 앱폼 공통 스키마/스캔 대비용 테이블만 유지.

## 5. TO-BE 설계서와의 관계

- **명의변경신청서_UI설계서_분석.md** 에서 정의한 흐름(가입유형 선택 → 고객(양도/양수) → 상품 → 동의 → 신청완료) 및 M플랫폼/KOS 연동은 **현재 AS-IS에 구현되어 있지 않음**.  
- AS-IS는  
  - 양도인/양수인 단계별 신청·조회·상태관리(포털 CsTransfer + 관리자),  
  - 앱폼 공통 저장 시 MCP_REQUEST_CHANGE 빈 값 INSERT 및 스캔 시 NAME_CHANGE_* 활용 가능 구조  
만 존재.  
- TO-BE에서 “명의변경 신청서”를 하나의 서식 플로우로 도입할 경우,  
  - 기존 NMCP_NFL_CHG_TRNS/TRNSFE 플로우와 통합할지,  
  - MCP_REQUEST_CHANGE를 실제 명의변경 입력값으로 채우고 M플랫폼/KOS와 연동할지  
를 정한 뒤, 컨트롤러/URL/매퍼 namespace(CsTransfer_Query vs CsTransferMapper) 정리 및 포털·관리자 역할 재정의가 필요함.
