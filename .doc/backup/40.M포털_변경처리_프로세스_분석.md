# M포털 변경처리 프로세스 구조 분석

기존 MCP(Mobile Customer Portal) 프로젝트의 변경처리 관련 프로세스를 분석한 문서입니다.

---

## 1. 개요

M포털의 "변경처리"는 크게 두 가지 유형으로 구분됩니다.

| 구분 | 업무 | 진입 | 처리 방식 |
|------|------|------|-----------|
| **신청서 기반** | 기기변경(HCN3), 우수기변(HDN3) | 요금제 설계·오픈마켓 | appForm 신청서 흐름 |
| **마이페이지 기반** | 요금상품 변경, 부가서비스 신청/해지, 유무선 결합 | 마이페이지 메뉴 | 즉시 M플랫폼 API 호출 |

스마트서식지에서 담당하는 **서비스변경 신청서(S102)** 는 위와 다른 별도 플로우이며, M포털의 기기변경·마이페이지 서비스변경 로직을 **참고**하여 신규 개발하는 구조입니다.

---

## 2. 전체 아키텍처

```
[브라우저]
    │
    │  (PC) /portal/appForm/...
    │  (모바일) /m/appForm/...
    ▼
[mcp-portal-web]  ← 정적 리소스 (JSP, JS, CSS)
    │
    │  AJAX: /appForm/saveAppformAjax.do, /mypage/regSvcChgAjax.do 등
    ▼
[mcp-portal-was]  ← Spring MVC 백엔드
    │
    │  MplatFormService
    │  MplatFormServerAdapter.callService()
    ▼
[M플랫폼]  ← KT 인프라 (X01, X20, X21, X38, X19 등)
    │
    ▼
[KT(KOS)]  ← 전사 시스템
```

- **mcp-portal-was**: appform, mypage, content 등 패키지에서 변경 관련 처리
- **mcp-portal-web**: `js/portal/appForm/appForm.js`, `js/portal/content/`, `js/portal/mypage/` 등에서 프론트 로직
- **mcp-api**: 일부 appform·mplatform API를 REST로 노출 (포털이 내부 호출)

## 3. 업무구분(OPER_TYPE) 코드

| 코드 | 명칭 | 설명 |
|------|------|------|
| NAC3 | 신규개통 | 유심/단말 신규 가입 |
| MNP3 | 번호이동 | 타사 회선 번호이동 |

- 소스: `mcp/mcp-portal-was/src/main/java/com/ktmmobile/mcp/common/constants/Constants.java`

## 4. 신청서 기반 변경처리 (기기변경·우수기변)

### 4.1 진입 경로

```
1) 요금제 설계 화면 (appFormDesignView)
   - /appForm/appFormDesignView.do
   - 파라미터: operType, hndsetModelId, instNom, rateCd, sprtTp 등
   - insertAppFormTempSave.do → requestKey 생성

2) 오픈마켓 (marketRequest)
   - /request/marketRequest.do (또는 /m/request/marketRequest.do)
   - 파라미터: a(조직ID), o(가입유형), s(판매정책), p(제품ID) 등
   - appOutForm.jsp로 리다이렉트

3) 신청서 본 화면 (appForm)
   - /appForm/appForm.do?requestKey={key}
   - appformReqTemp 검증 후 appFormDesignView.do로 리다이렉트
   - JSP: portal/appForm/appForm.jsp (또는 mobile)
   - JS: appForm.js
```

### 4.2 5단계 신청서 플로우 (appForm.js)

| Step | 내용 | operType=CHANGE/EXCHANGE 시 |
|------|------|-----------------------------|
| 1 | 본인확인·약관동의 | updateRequestTemp.do |
| 2 | 유심정보·신분증 확인 | - |
| 3 | 가입신청 정보 (할부, 납부 등) | updateRequestTemp3.do |
| 4 | 부가서비스 정보 | **fnSaveForm() 호출 → step5 생략** |
| 5 | 납부정보·가입정보 확인 | (기기변경·우수기변은 미진입) |
| 6 | (셀프개통 시) 개통요청 | - |

- 기기변경·우수기변: step4 완료 시점에 `fnSaveForm()` 호출하여 신청서 저장 후 완료
- 신규·번호이동: step5까지 진행 후 `fnSaveForm()` 호출

### 4.3 기기변경 전용 인증

- **SessionUtils.getChangeAutCookieBean()**: 기기변경 시 사용하는 고객 인증 세션
- `saveAppformAjax.do`에서 operType이 HCN3/HDN3일 때:
  - ChangeAutCookieBean(ctn, 생년월일, contractNum) 검증
  - 검증 실패 시 `CHANGE_AUT_EXCEPTION` 발생

### 4.4 주요 API·컨트롤러

| 기능 | 엔드포인트 | 소스 |
|------|------------|------|
| 임시저장 생성 | POST /appForm/insertAppFormTempSave.do | AppformController.insertAppFormTempSave() |
| Step1 임시저장 | POST /appForm/updateRequestTemp.do | AppformController.updateRequestTemp() |
| Step3~5 임시저장 | POST /appForm/updateRequestTemp3~5.do | AppformController |
| 부가서비스 임시목록 | /appForm/getAdditionTempListAjax.do | AppformController.getAdditionTempList() |
| 신청서 최종 저장 | POST /appForm/saveAppformAjax.do | AppformController.saveAppformDb() |
| 개통사전체크 | POST /appform/conPreCheckAjax.do | AppformController.conPreCheck() |
| 요금제 설계 화면 | /appForm/appFormDesignView.do | AppformController.appformDesignView() |
| 신청서 본 화면 | /appForm/appForm.do | AppformController.appform() |
| 신청서 완료 | /appForm/appFormComplete.do | AppformController.appFormComplete() |
| 오픈마켓 진입 | /request/marketRequest.do | AppformController.marketRequest() |

## 5. 마이페이지 기반 서비스변경

### 5.1 요금상품 변경

- **진입**: `/mypage/farPricePlanView.do` (또는 `/m/mypage/farPricePlanView.do`)
- **컨트롤러**: `FarPricePlanController`
- **서비스**: `FarPricePlanService` → `MplatFormService.farPricePlanChgNeTrace()`, `regSvcChgNeTrace()`
- **M플랫폼 API**: X19(요금상품 변경), X21(부가서비스 신청)
- **프론트**: `mcp-portal-web/src/resources/js/portal/mypage/farPricePlan*.js` 등

### 5.2 부가서비스 신청/해지

- **진입**: `/mypage/regServiceView.do`
- **컨트롤러**: `RegSvcController`
- **엔드포인트**:
  - `POST /mypage/regSvcChgAjax.do`: 부가서비스 신청 (X21)
  - `POST /mypage/regSvcCanChgAjax.do`: 부가서비스 해지 (X38)
- **서비스**: `RegSvcService` → `MplatFormService.regSvcChg()`, `moscRegSvcCanChgNeTrace()`
- **프론트**: `mcp-portal-web/src/resources/js/portal/mypage/regService*.js` 등

### 5.3 유무선 결합

- **진입**: `/content/myCombinationView.do`
- **컨트롤러**: `MyCombinationController`
- **엔드포인트**:
  - `POST /content/insertCombinationAjax.do`: 결합 저장
  - `GET /content/combinationCheckAjax.do`: 결합 가능 여부 조회
- **M플랫폼 API**: X78(결합 사전체크), X79(결합 저장)
- **참고**: `KT_M플랫폼_X78_X79_연동명세.md`

### 5.4 부가서비스·가입정보 조회 (content)

- **컨트롤러**: `MyCombinationController`, `content` 패키지
- **M플랫폼 API**: X20(부가서비스 조회) → `MplatFormService.moscCombInfo()`
- **소스**: `mcp/mcp-portal-was/src/main/java/com/ktmmobile/mcp/content/controller/MyCombinationController.java`

## 6. M플랫폼 API 연동 (변경처리 관련)

### 6.1 호출 구조

```
MplatFormService
    └─ MplatFormServerAdapter.callService(param, vo)
         └─ HttpClientUtil → M플랫폼 URL
         └─ vo.toResponseParse() → XML 파싱
```

### 6.2 변경처리 관련 이벤트코드

| 이벤트코드 | 용도 | 호출 메서드 (MplatFormService) |
|------------|------|-------------------------------|
| X01 | 가입정보 조회 | perMyktfInfo() |
| X02 | 청구지주소 변경 | perAddrChg() |
| X04 | e-mail청구서 신청/변경/해지 | bilEmailChg() |
| X19 | 요금상품 변경 | farPricePlanChgNe() |
| X20 | 부가서비스 조회 | moscCombInfo() |
| X21 | 부가서비스 신청 | regSvcChg(), regSvcChgNe(), regSvcChgNeTrace() |
| X38 | 부가서비스 해지 | moscRegSvcCanChg(), moscRegSvcCanChgNeTrace() |
| X78 | 유무선 결합 사전체크 | moscCombChkInfo() |
| X79 | 유무선 결합 저장 | moscCombSaveInfo() |
| X89 | 요금제 예약변경 조회 | moscFarRsvChgInqr() |

### 6.3 주요 VO/DTO

- `MpPerMyktfInfoVO`: X01 가입정보
- `MpAddSvcInfoDto`, `MpSocVO`: X20 부가서비스
- `MpRegSvcChgVO`, `RegSvcChgRes`: X21/X38 부가서비스 신청/해지

## 8. 데이터 흐름 요약

### 8.1 기기변경(HCN3) 저장 시퀀스

```
1. appForm.js step4 [다음] 클릭
   → fnSaveTempStep4()  // updateRequestTemp4.do
   → fnSaveForm()       // saveAppformAjax.do

2. AppformController.saveAppformDb()
   → SessionUtils.getChangeAutCookieBean() 검증 (HCN3/HDN3)
   → appformSvc.saveAppform()
      → MCP_REQUEST 등 테이블 INSERT
      → OSST 연동 (개통 프로세스)

3. appFormComplete.do → 완료 화면
```

### 8.2 부가서비스 신청 (마이페이지)

```
1. regServiceView.do → 부가서비스 화면
2. myAddSvcListAjax.do → X20 현재 부가서비스 조회
3. regSvcChgAjax.do → X21 부가서비스 신청
   → RegSvcService.regSvcChg()
   → MplatFormService.regSvcChg() / regSvcChgNeTrace()
4. 결과 반환 (JSON)
```

### 8.3 요금상품 변경 (마이페이지)

```
1. farPricePlanView.do → 요금상품 변경 화면
2. farPricePlanChgAjax.do → X19 요금상품 변경
   → FarPricePlanService
   → MplatFormService.farPricePlanChgNeTrace()
3. 결과 반환
```

## 9. 스마트서식지 개발 시 참고 포인트

- **서비스변경 신청서(S102)** 는 M포털 appForm과 **다른 플로우**이며, mform-web에서 **신규 개발**합니다.
- 참고할 M포털 소스:
  - 가입정보/부가서비스 조회: `MplatFormService` X01, X20
  - 부가서비스 신청/해지: X21, X38
  - 요금상품 변경: X19
  - 동시 변경 불가 체크: appForm.js 내 operType·step 분기, conPreCheck 등
- 연동 API 스펙은 M포털과 **동일하게 구현**하며, 코드는 재사용이 아닌 **동일 스펙 신규 구현**입니다.
