# 명의변경 관련 M전산 소스 분석

**목적**: 명의변경(ident) 플로우에서 사용하는 M전산 조회·연동 소스 정리  
**참조**: `45.휴대폰인증_Y04_계약정보유효성_확인.md`, `51.명의변경_ASIS_소스분석.md`, `52.명의변경_TOBE_개발분석.md`

---

## 1. 개요

명의변경에서 **M전산**이 쓰이는 곳은 다음 두 가지다.

| 구분 | 용도 | M전산 역할 |
|------|------|------------|
| **휴대폰 인증(join-info)** | 양도고객 이름+휴대폰으로 본인 확인 후 ncn/custId 확보 | 이름+휴대폰 → 계약번호(ncn)·고객번호(custId) 조회 |
| **법정대리인 SMS 인증** | 양도/양수 미성년자 시 법정대리인 인증 | 설계서상 M전산 모듈로 인증번호 발송/확인 (미구현) |
| **신청 후 후속 처리** | 접수된 명의변경 건에 대해 가족관계 서류 확인 후 명의변경 실행 | 고객센터/상담사 확인 후 **M전산에서** 가족관계 서류 확인·명의변경 처리 (설계서: IT 프로세스 검토 예정) |

이 문서는 **1) 휴대폰 인증**에 쓰이는 M전산 조회 소스를 중심으로 정리하며, **신청 후 M전산에서 가족관계 서류 확인·처리** 흐름은 §8에서 정리한다.

---

## 2. M전산 조회 목적 (명의변경 휴대폰 인증)

- **입력**: 고객이 입력한 **이름(name)** + **휴대폰번호(ctn)**  
- **출력**: **ncn(계약번호)**, **custId(고객번호)**  
- **이유**: M플랫폼 Y04(계약정보 유효성체크) 호출 시 `ncn`, `ctn`, `custId`가 필수인데, `ncn`·`custId`는 고객이 모르는 값이라 M전산에서 이름+휴대폰으로 조회해야 함.

**흐름 요약**

1. 고객: 이름, 휴대폰 입력 → [인증]  
2. 백엔드: **M전산 조회** (이름 + 휴대폰 → ncn, custId)  
3. 백엔드: **Y04** 호출 (clientIp, cIntUsrId, ncn, ctn, custId)  
4. 성공 시: JoinInfoResponse에 success, ncn, custId, addr, email 등 반환 → 이후 단계(부가서비스, 요금제 등)에서 ncn/custId 사용

## 3. 스마트서식지(mform-api) – M전산 연동

### 3.1 호출 경로 (명의변경)

- **API**: `POST /api/v1/ident/join-info`  
- **컨트롤러**: `com.ktmmobile.msf.formOwnChg.controller.OwnChgRestController#identJoinInfo`  
- **서비스**: `OwnChgInfoSvc` → `SvcChgInfoService.joinInfo()` (formSvcChg 공통)

### 3.2 M전산 조회 계층 (formSvcChg 공통)

| 계층 | 클래스 | 파일 | 역할 |
|------|--------|------|------|
| Service | SvcChgInfoServiceImpl | formSvcChg/service/SvcChgInfoServiceImpl.java | name, ctn 검증 → M전산 조회 → Y04 호출 → 응답 생성 |
| Mapper | SvcChgMypageMapper | formSvcChg/mapper/SvcChgMypageMapper.java | selectContractInfo(name, mobileNo) 인터페이스 |
| Mapper XML | SvcChgMypageMapper.xml | resources/mapper/SvcChgMypageMapper.xml | M전산(또는 로컬 테이블) 조회 쿼리 |
| DTO | ContractInfoDto | formSvcChg/dto/ContractInfoDto.java | ncn, custId 담는 조회 결과 |

### 3.3 SvcChgInfoServiceImpl – M전산 조회 사용 부분

**파일**: `mform-api/src/main/java/com/ktmmobile/msf/formSvcChg/service/SvcChgInfoServiceImpl.java`

- `joinInfo(JoinInfoRequest req, String clientIp)` 내부:
  1. `req.getCtn()`, `req.getName()` 검증
  2. `params.put("name", name.trim())`, `params.put("mobileNo", ctn)` (숫자만 정리한 ctn)
  3. **`svcChgMypageMapper.selectContractInfo(params)`** → `ContractInfoDto` (ncn, custId)
  4. M전산 조회 실패 시:  
     - `SERVER_NAME=LOCAL` 이고 `mform.join-info.mock-when-no-db=true` 이면 Mock ncn/custId 로 Y04까지 진행  
     - 아니면 "고객 정보와 휴대폰번호가 일치하지 않습니다" 반환
  5. `contractInfo.getNcn()`, `contractInfo.getCustId()` 로 **Y04** 호출:  
     `mplatFormSvc.contractValdChk(clientIp, cIntUsrId, ncn, ctn, custId)`
  6. 성공 시 `JoinInfoResponse`에 success, ncn, custId, addr, email 등 세팅 후 반환

### 3.4 SvcChgMypageMapper.xml – 쿼리 (M전산/로컬)

**파일**: `mform-api/src/main/resources/mapper/SvcChgMypageMapper.xml`

```xml
<select id="selectContractInfo" parameterType="map" resultType="com.ktmmobile.msf.formSvcChg.dto.ContractInfoDto">
    SELECT contract_num AS ncn,
           customer_id AS custId
      FROM msp_juo_sub_info
     WHERE sub_status <> 'C'
       AND TRIM(subscriber_no) = TRIM(#{mobileNo})
       AND UPPER(REPLACE(COALESCE(sub_link_name, ''), ' ', '')) = UPPER(REPLACE(COALESCE(#{name}, ''), ' ', ''))
</select>
```

- **로컬(개발)**: PostgreSQL `msp_juo_sub_info` 테이블 (kcf_tab_create1.sql 실행으로 생성)
- **운영**: Oracle M전산 사용 시 `MSP_JUO_SUB_INFO@DL_MSP` 등으로 FROM 절 교체, 컬럼명(CONTRACT_NUM, CUSTOMER_ID, SUBSCRIBER_NO, SUB_LINK_NAME, SUB_STATUS) 맞추면 됨

### 3.5 ContractInfoDto

- **ncn**: 사용자 서비스 계약번호 (M전산 CONTRACT_NUM)
- **custId**: 고객번호 (M전산 CUSTOMER_ID)

## 4. M포탈(MCP) – M전산 조회 참고

명의변경 전용이 아니라, **비회원 휴대폰 인증(회선 등록 등)** 에서 동일한 “이름+휴대폰 → 계약번호” 조회를 쓴다.

### 4.1 MCP API – 계약번호 조회

| 항목 | 내용 |
|------|------|
| API | `POST /mypage/contractNum` |
| Controller | mcp-api `com.ktmmobile.mcp.mypage.controller.MypageController#contractNum` |
| 파라미터 | `HashMap`: `name`, `mobileNo` |
| 반환 | `String` (계약번호 CONTRACT_NUM만, custId 없음) |

### 4.2 MCP MypageMapper.xml – M전산 쿼리

**파일**: `mcp/mcp-api/src/main/resources/mcp/mapper/MypageMapper.xml`

```xml
<select id="selectContractNum" parameterType="java.util.Map" resultType="java.lang.String">
    SELECT CONTRACT_NUM, SUBSCRIBER_NO AS MOBILE_NO
      FROM MSP_JUO_SUB_INFO@DL_MSP
     WHERE SUB_STATUS <![CDATA[<>]]> 'C'
       AND SUBSCRIBER_NO = TRIM(#{mobileNo})
       AND UPPER(REPLACE(SUB_LINK_NAME,' ','')) = UPPER(REPLACE(#{name},' ',''))
</select>
```

- **DL_MSP**: M전산 DB 링크
- **MSP_JUO_SUB_INFO**: M전산 가입자 정보 (SUBSCRIBER_NO=휴대폰, SUB_LINK_NAME=이름, CONTRACT_NUM=계약번호)
- MCP는 반환 타입이 String이라 **CONTRACT_NUM(ncn)만** 반환. custId가 필요하면 동일 테이블/뷰에서 CUSTOMER_ID 등 컬럼 추가 조회 필요.

### 4.3 MCP WAS – 비회원 휴대폰 인증 시 계약번호 사용

**파일**: `mcp-portal-was/.../mypage/controller/MypageUserController.java` (약 806~859행)

- 이름(name), 휴대폰(phone)으로 **MypageUserService.selectContractNum(name, phone)** 호출  
  → 내부에서 MCP API `/mypage/contractNum` 호출 → M전산 조회
- 계약번호 없으면: RESULT_CODE "0002", "사용자 정보가 일치하지 않습니다"
- 계약번호 있으면: 선납 여부 등 추가 검사 후 `rtnMap.put("contractNum", contractNum)` 로 반환
- 이후 해당 요청/세션에서 **ncn(contractNum)** 을 가지고 Y04 등 M플랫폼 연동 가능

**파일**: `mcp-portal-was/.../mypage/service/MypageUserServiceImpl.java`

- `selectContractNum(String name, String phone)`: RestTemplate으로 `apiInterfaceServer + "/mypage/contractNum"` 호출, map에 `name`, `mobileNo` 담아 전달

## 5. M전산 테이블/뷰 정리

| 소스 | 테이블/뷰 | 링크/스키마 | 주요 컬럼 (조회 조건·결과) |
|------|-----------|-------------|---------------------------|
| MCP | MSP_JUO_SUB_INFO | @DL_MSP | SUBSCRIBER_NO(휴대폰), SUB_LINK_NAME(이름), SUB_STATUS, CONTRACT_NUM(ncn) |
| mform-api (로컬) | msp_juo_sub_info | PostgreSQL 로컬 | subscriber_no, sub_link_name, sub_status, contract_num(ncn), customer_id(custId) |
| mform-api (운영 가정) | MSP_JUO_SUB_INFO | @DL_MSP 등 | CONTRACT_NUM(ncn), CUSTOMER_ID(custId), SUBSCRIBER_NO, SUB_LINK_NAME, SUB_STATUS |

- **SUB_STATUS <> 'C'**: 해지 등 비정상 상태 제외
- 이름/휴대폰 비교 시 공백 제거 후 대소문자 구분 없이 비교 (UPPER, REPLACE)

## 6. 명의변경 전용 플로우(AS-IS)와 M전산

- AS-IS 명의변경 전용(양도인/양수인)은 **CsTransferSvc**, **NMCP_NFL_CHG_TRNS/TRNSFE** 테이블 사용.
- **M전산 조회**는 “명의변경 전용”이 아니라 **휴대폰 인증(이름+휴대폰 → ncn/custId)** 용으로, MCP에서는 비회원 인증/회선 등록 등에서 사용하고, 스마트서식지에서는 **ident/join-info** 에서 동일한 M전산 조회 로직(SvcChgInfoService + SvcChgMypageMapper)을 공유한다.

## 7. 정리

| 구분 | M전산 조회 | Y04 호출 | 비고 |
|------|------------|----------|------|
| **명의변경(TO-BE)** | OwnChgRestController → SvcChgInfoService → SvcChgMypageMapper.selectContractInfo | MplatFormSvc.contractValdChk | formOwnChg는 진입만, 실제 조회·Y04는 formSvcChg 공통 |
| **조회 쿼리** | msp_juo_sub_info (로컬) / MSP_JUO_SUB_INFO@DL_MSP (운영) | - | name, mobileNo → ncn, custId |
| **MCP 참고** | MypageMapper.selectContractNum, MypageController.contractNum | - | ncn만 반환, custId는 미조회 |

- **법정대리인 SMS 인증**: 설계서상 M전산 모듈 사용 예정이나, 현재 스마트서식지·MCP 명의변경 소스에는 해당 호출 미구현.

## 8. 명의변경 신청 후 M전산에서 가족관계 서류 확인·처리

### 8.1 설계서상 흐름

UI 설계서(50.명의변경신청서_UI설계서_분석.md) 기준:

- **신청 시**: 신청서 작성 완료 → [작성 완료] → MC0 명의변경사전체크 → MP0 명의변경 → 리포트 이미징 → 이미징 시스템 전송 → **K-NOTE(신청서·첨부서류)**.
- **후속**: **신청서·첨부서류 전송으로 접수만 진행**, **고객센터/상담사 확인 후 M전산에서 후속 처리**(IT 프로세스 검토 예정).

즉, 앱/웹에서는 “접수”까지만 하고, **가족관계 서류 확인 후 실제 명의변경 반영**은 **M전산 측**에서 하는 업무로 설계되어 있다.

### 8.2 “가족관계 서류 확인 후 명의변경 처리” 위치

| 구분 | 내용 |
|------|------|
| **의도** | 명의변경 신청 접수 후, M전산에서 **가족관계 서류**(가족관계증명서 등)를 확인한 뒤 명의변경을 처리한다. |
| **현재 코드베이스** | **이 리포지토리(mform, MCP)에는** “M전산에서 가족관계 서류를 확인한 뒤 명의변경 처리한다”는 **직접적인 호출·배치·화면이 없음**. |
| **관리자 기능** | MCP 관리자(mcp-admin-was) **명의변경** 메뉴는 **상태값(TRNS_STATUS_VAL)** 과 **확인메모(CONFIRM_MEMO)** 만 저장한다. |

**관리자 소스 (참고)**

| 항목 | 경로 | 역할 |
|------|------|------|
| 리스트/상세 | `mcp-admin-was/.../cs/controller/TransferController.java` | `transferList`, `transferDetail` – NMCP_NFL_CHG_TRNS/TRNSFE 조회 |
| 상태·메모 저장 | `transferDetailSave.do` → `TransferSvc.transferDetailSave` | `CsTransferMapper.transferDetailSave` – TRNS_STATUS_VAL, CONFIRM_MEMO, AMD_ID, AMD_DT 만 UPDATE |
| 개인정보 삭제(처리완료) | `transferDetailDel.do` → `TransferSvc.transferDetailDel` | 개인정보 마스킹 후 TRNS_STATUS_VAL='3'(처리완료) 로 UPDATE. **M전산/OSST 호출 없음** |

따라서 **“가족관계 서류 확인”** 및 **“M전산에서 명의변경 처리 실행”**은 다음 중 하나로 추정된다.

1. **M전산 전용 업무**: M전산 시스템(별도 배치·화면)에서 NMCP_NFL_CHG_* 또는 연계 테이블/인터페이스를 조회한 뒤, 서류 확인 후 명의변경 처리(예: OSST/MP0 연동)를 수행.
2. **수동 프로세스**: 상담사가 K-NOTE 등에서 서류 확인 후, 관리자 화면에서 “처리완료”로 변경하거나, M전산 쪽에 별도 입력/배치를 돌리는 방식.

### 8.3 M플랫폼 MP0(명의변경 처리)와의 관계

- **MP0**: M플랫폼/OSST 측 **명의변경 실행** 오퍼레이션 (`osst:osstCanM2mPrc`, `OsstCanSvcPrcService`).
- **mcp-api** `MPlatformMapper.getRequestInfo` 에서 `appEventCd = 'MP0'` 일 때 위 서비스/오퍼레이션명을 반환하며, **실제 MP0 호출 시 필요한 XML 메시지**는 msc-prx 등에서 조합한다.
- 즉, **“언제·어디서” MP0를 호출할지**(가족관계 서류 확인 완료 후 M전산에서 호출하는지 등)는 **이 코드베이스에는 구현되어 있지 않으며**, M전산·M플랫폼 연동 규격 및 운영 프로세스 확인이 필요하다.

### 8.4 정리

| 항목 | 내용 |
|------|------|
| **가족관계 서류 확인** | 설계서상 “고객센터/상담사 확인 후 M전산에서 후속 처리”에 포함되는 업무로 해석되나, **mform·MCP 소스에는 해당 확인을 수행하거나 M전산을 호출하는 코드 없음**. |
| **명의변경 처리 실행** | 관리자 화면은 **상태/메모만 변경**. 실제 명의변경 반영(MP0 등)은 **M전산 측 배치·업무**에서 수행하는 것으로 보이며, 그 부분은 본 리포지토리 밖에 있음. |
| **추가 확인 필요** | M전산에서 가족관계 서류 확인·명의변경 처리하는 **배치명·화면·인터페이스**는 M전산/운영 문서 또는 담당자 확인 필요. |
