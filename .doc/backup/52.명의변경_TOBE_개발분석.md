## 1. 개요

### 1.1 TO-BE 구조

| 구분 | 내용 |
|------|------|
| **프론트엔드** | mform-web (Vue3) |
| **백엔드** | mform-api (Spring Boot) – formSvcChg 공유 |
| **플로우** | 가입유형(대리점) → 고객 → 상품 → 동의 → 신청완료 |
| **메뉴** | msf_menu.js id='03' 명의변경, stepSubMenu |

### 1.2 설계서 vs AS-IS vs TO-BE

| 항목 | 설계서(S1030*) | AS-IS | TO-BE |
|------|----------------|-------|-------|
| **플로우** | 고객→상품→동의→신청완료 | 양도인 신청 → 양수인 신청 (별도 플로우) / 앱폼 MCP_REQUEST_CHANGE 빈값 | 설계서 플로우(단일 서식) |
| **진입 화면** | 명의변경 타입+고객 통합 | - | IdentType 제거, IdentTypeCust(가입유형·고객) 1단계로 통합 |
| **데이터 저장** | M플랫폼·이미징 전송 | NMCP_NFL_CHG_TRNS/TRNSFE, MCP_REQUEST_CHANGE | **POST /ident/apply** 연동 완료. 저장 시 백엔드 검증·접수번호 반환. MC0/MP0·DB 저장 추후 |

## 4. AS-IS 대비 TO-BE 차이

### 4.1 플로우

| 구분 | AS-IS | TO-BE |
|------|-------|-------|
| 양도인/양수인 | 단계 분리 (양도인 신청 → 양수인 조회·등록) | 단일 서식 (고객→상품→동의) |
| 테이블 | NMCP_NFL_CHG_TRNS, NMCP_NFL_CHG_TRNSFE | 미사용 (store 기반, 백엔드 저장 미구현) |
| MCP_REQUEST_CHANGE | 앱폼 저장 시 빈값 INSERT | 미연동 |

### 4.2 휴대폰 인증

| 구분 | AS-IS | TO-BE |
|------|-------|-------|
| 방식 | NICE 본인인증 (param_r3='W'), 세션 OTP | join-info (이름+ctn) → M전산 조회 → Y04 |
| 연동 | NiceCertifyController | SvcChgInfoService (방안 A) |

### 4.3 데이터 저장

| 구분 | AS-IS | TO-BE |
|------|-------|-------|
| 저장 시점 | 양도인 grantorRegSave, 양수인 assigneeRegSave | **save() 시 submitIdentApply(store) → POST /ident/apply → 접수번호 반환 후 reset·완료** |
| 관리자 | transferList, transferDetail, 처리 | mcp-admin-was 기존 화면. mform 저장 데이터 연동 추후 |
