# 스마트서식지 MCF — 프로젝트 구조 및 기능분석 통합 문서

본 문서는 아래 세 문서를 기반으로 **현재 MCF 구현 구조와 기능분석 전체 기능을 한 번에 정리**한 통합 문서이다.

- `30.개발기능목록_담당.md`
- `15.개발기능목록_ASIS_TOBE_기능분석명세서.md`
- 기존 `16.MCF_프로젝트_기능분석명세서.md`

목적은 다음과 같다.

- 현재 `mform-web`, `mform-api` 패키지 구조를 기준으로
  서비스변경/명의변경/서비스해지 신청서의 전체 기능을 어디까지 수용했는지 정리
- 아직 구현되지 않은 기능은 **Controller / Service / 메서드 이름까지 제안**하여
  추후 개발 시 기준이 되는 “깔끔한 프로젝트 구조”를 제시

## 6. 기능분석 전체 기능 vs 현재 구조 적용성 평가

본 절은 `30.개발기능목록_담당.md` 의 **모든 기능 항목(서비스변경/명의변경/서비스해지)** 을
현재 MCF 패키지 구조에 어떻게 대응시킬 수 있는지, 그리고 그 적용성이 어떤지 정리한 것이다.

### 6.1 서비스변경 신청서 기능군

#### 6.1.1 고객·상품·동의 3단계 (REQ_010201, 010903, 010906)

- **기능분석 요약**
  - S102010101, S102020101, S102030101, S102040101, S102050101 등  
    → 서비스변경 신청서의 메인/고객/상품/동의/신청완료/미리보기/열람 비밀번호 입력 등 3단계 UI.
- **현재 구조 적용**
  - Front: `McpServiceDetailView.vue` + `ChangeTypeCust.vue` + `ChangeProd.vue` + `ChangeAgree.vue`
  - Back:
    - 공통: `formComm.SvcChgRestController` + `JoinInfoSvc` (join-info, 계약정보)
    - 도메인별: `SvcChgBaseSvc`(업무코드), `SvcChgRegSvc`(부가), `FarPriceSvc`(요금제),
      `SvcChgPauseSvc`, `SvcChgDataSharingSvc`, `SvcChgCombineSvc` 등
- **평가**
  - 기능분석의 “3단계 구조 + 영역별 세부 기능” 모델이 현재 MCF 구조에 그대로 녹아 있음.
  - `30번 문서` 의 화면번호/상세기능/연동대상은 대부분 **이미 구현된 Controller/Service 메서드와 1:1 또는 N:1 매핑** 가능.

#### 6.1.2 부가서비스/요금제/번호변경/LOST_RESTORE (REQ_010202~010207)

- **부가서비스/요금제**
  - X20/X21/X38, Y24/Y25, X89/X90 관련 기능들은
    - `SvcChgRegSvc`, `FarPriceSvc`, `MplatFormSvc` 내 메서드로 구현되어 있어  
      기능분석 상의 API 매핑을 그대로 반영 가능.
  - 화면 구조는 `ChangeProd.vue` (상품 영역) + 부가/요금제 팝업(`McpAdditionEditPop` 등)으로 대응.
- **번호변경(NU2/X32)**
  - 기능분석에는 신규번호 검색·조회횟수카운트·번호변경이 상세 정의되어 있으나,
  - MCF는 아직 별도 `SvcChgNumber*` 계층이 없어 **1장 설계안(SvcChgNumberController/SvcChgNumberSvc)** 를 통해 추가 설계·구현 필요.
- **LOST_RESTORE(X26/X28/X30/X33/X35)**
  - 기능분석(47번 문서)와 MCF 구현(`SvcChgPauseController/SvcChgPauseSvc`) 구조가 일치.
  - 기능분석의 처리흐름을 현재 서비스에 그대로 적용한 상태로 봐도 무방.

#### 6.1.3 데이터쉐어링/아무나 SOLO (REQ_010211, 010212)

- **데이터쉐어링(X69/X70/X71)**
  - 기능분석(43번 문서)의 사전체크/가입대상 조회/가입·해지 로직이  
    `SvcChgDataSharingSvc` 의 `check/list/join/cancel` 메서드로 구현되어 있어 적용성이 높음.
  - `ChangeProd.vue` 의 데이터쉐어링 카드에서 이 API 집합을 그대로 사용 가능.
- **아무나 SOLO(Y44, X20/X87, PL249Q800)**
  - 기능분석(44,45번 문서)의 체크/신청 흐름이  
    `SvcChgCombineSvc` + `SvcChgCombineMapper` + `MplatFormSvc` 구조로 옮겨져 있음.
  - 기능분석상 세부 예외코드/메시지도 향후 `SvcChgCombineSvcImpl` 에 단계적으로 반영 가능.

#### 6.1.4 단말보험·USIM·서식지/이미징/전송 (REQ_010208, 010210, 010213, 010904~010906)

- **단말보험/USIM**
  - 기능분석에는 상세 기능과 연동 요구사항이 정의되어 있으나  
    MCF 패키지에 `SvcChgInsr*`, `SvcChgUsim*` 계층이 아직 없어 **미구현**.
  - 1장에서 제안한 이름(`SvcChgInsrController/SvcChgInsrSvc`, `SvcChgUsimController/SvcChgUsimSvc`)으로
    추가하면 기능분석 문서와의 매핑이 명확해짐.
- **서식지/이미징/전송/알림**
  - 서비스변경/명의변경/해지 모두 기능분석에는 IF_0030/0031/0035/0038 흐름이 있으나  
    MCF에는 공통 전송 모듈이 없는 상태.
  - `FormSendController/FormSendSvc` 공통 구조를 도입하면,
    전송/이미징 기능을 세 도메인에 일관되게 적용할 수 있음.

### 6.2 명의변경 신청서 기능군

#### 6.2.1 메인/고객/상품/동의/완료 (REQ_010301, 010302, 010903, 010906)

- **현재 적용**
  - Front: `IdentTypeCust.vue`, `IdentCust.vue`, `IdentProd.vue`, `IdentAgree.vue`
  - Back: `OwnChgController` + `OwnChgInfoSvc/OwnChgAddSvc/OwnChgApplySvc` + `OwnChgMapper`
- **평가**
  - 기능분석의 화면/단계 구조가 MCF 구현과 거의 1:1 로 일치.
  - 신청서 작성/저장/완료/열람 기능은, 서비스변경과 마찬가지로  
    공통 `FormSend*` 구조 도입 시 연계가 자연스럽다.

#### 6.2.2 신용정보/납부정보/계좌·카드/청구ID/신청서 이미징 (REQ_010301, 010904~010906)

- 기능분석에 정의된 NICE(IF_0033, 0016, 0017)·청구ID·이미징/전송 기능은
  현재 MCF 구현이 없어 `CreditCheckController/CreditCheckSvc`, `FormSendController/FormSendSvc`,
  `RecordingController/RecordingSvc` 등을 공통 계층으로 추가해야 한다.
- 이 공통 계층은 서비스변경·명의변경·해지에서 함께 사용할 수 있어
  **패키지 구조 관점에서 중복을 줄이고 기능분석 요구사항을 일관되게 수용**할 수 있다.

### 6.3 서비스해지 신청서 기능군

- **현재 적용**
  - Front: `CancelType.vue`, `CancelCust.vue`, `CancelProd.vue`, `CancelAgree.vue`
  - Back: `CancelController` + `CancelApplySvc` (신청/해지 처리 골격)
- **평가**
  - 기능분석 상 해지 3단계 구조는 이미 화면 구조에 반영되어 있으며,
  - 잔여요금/위약금/분납금/사전체크/이미징/전송/알림 기능만 추가하면
    서비스변경/명의변경과 유사한 완성도를 가질 수 있다.

#### 6.3.2 잔여요금/위약금/분납금/사전체크 (REQ_010401)

- 기능분석에서 X22/X67 기반 요금·위약금·분납금 조회, 해지 사전체크 API가 정의되어 있으나,
  현재 MCF에는 전용 `CancelInfoController/CancelInfoSvc`, `CancelEligibilityController/CancelEligibilitySvc`
  가 없다.
- 3장에서 제안한 구조대로 구현할 경우,
  - 서비스해지 도메인이 서비스변경/명의변경과 동일한 수준의 M-Platform 연동 레이어를 갖게 되고,
  - 기능분석에서 정의한 **해지 전 사전 검증 흐름**을 그대로 적용할 수 있다.

### 6.4 종합 평가

- **구현이 완료된 영역**(서비스변경 주요 기능, 명의변경 핵심 플로우, 데이터쉐어링/LOST_RESTORE/아무나 SOLO 등)은  
  기능분석 문서의 구조와 MCF 패키지 구조가 잘 맞물려 있어,  
  `15/30번 문서` 의 TO-BE 컬럼에 현재 구현을 기준으로 상세 매핑만 해주면 된다.
- **미구현 영역**(번호변경, 단말보험, USIM 변경, 신용정보/납부정보, 서식지/이미징/전송, 해지 요금/사전체크 등)은  
  본 문서 1~3장에서 정의한 공통/도메인별 Controller·Service 계층을 추가하면,  
  기능분석의 모든 기능을 현재 MCF 구조에 **무리 없이 수용**할 수 있다.
- 특히 공통 계층(`MplatFormSvc`, `FormSendSvc`, `CreditCheckSvc`)을 활용하는 방향은  
  기능분석에서 도메인별로 반복되는 요구사항(요금/신용/전송/알림)을 **중복 없이 재사용**하게 해 주므로,  
  향후 유지보수와 확장성 측면에서도 유리하다.
