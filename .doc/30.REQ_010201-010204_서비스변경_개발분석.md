관련 기능번호: **C01, C02, C03, C07, C08, C09, C10, C11, C75**

#### 개발 대상 (백엔드)

**① formComm — 가입자정보조회 (C02, C03, C11)**

| 항목 | 내용 |

**③ formSvcChg — 동시 처리 불가 체크 (C08) — 미구현 항목**

| 항목 | 내용 |
|------|------|
| Controller | `SvcChgRestController` (또는 `SvcChgBaseController`) |
| 엔드포인트 | `POST /api/v1/service-change/concurrent-check` |

| 처리 흐름 | `selectedCodes`(선택한 업무 코드 목록) 입력 → 동시 불가 조합 룰 검증 → `valid(boolean)`, 불가 사유 반환 |
| ReqDto | `ConcurrentCheckReqDto` (selectedCodes: List\<String\>) |
| ResVO | `ConcurrentCheckResVO` (valid: boolean, message: String) |

동시 불가 조합 (ASIS 기준):
- 번호변경 + 분실복구/일시정지해제
- 번호변경 + 요금상품변경
- 번호변경 + 기변
- 이 조합을 코드 상수 또는 DB `MSF_CD_DTL` 에 정의

**④ 대리점/판매점 정보 조회 (C09, C75) — 미구현 항목**

대리점 정보는 로그인 세션 기반으로 제공. 현재 MSF에 로그인 세션 구조 미정의 → 향후 인증 모듈 구현 후 진행. 본 sprint 제외.

### 2.2 REQ_010202 — 무선데이터차단 서비스 (C12, C13, C14)

- `true`: "무선데이터 이용" 선택지만 표시 (차단 중이므로 이용으로 전환 가능)
- `false`: "무선데이터차단 서비스 이용" 선택지만 표시 (미사용이므로 차단 가입 가능)

### 2.3 REQ_010203 — 정보료 상한 (C15, C16, C17)

REQ_010202와 동일 `SvcChgRegController` / `SvcChgRegSvcImpl`에서 처리. X20 조회 결과에서 정보료 상한 SOC 여부를 판단하여 `infoLimitAmount` 필드로 반환. 별도 Controller 불필요.

- 화면에서 이용중/미이용에 따라 옵션 분기

### 2.4 REQ_010204 — 부가 신청/변경 (C18~C20, C46~C49)

REQ_010202와 동일 Controller/Service 재사용. 추가 개발 포인트는 설정 팝업(상품파람정보) 연동.

**추가 팝업 화면 설정 API (상품파람정보, ftrNewParam):**

| 팝업 | SOC 예시 | ftrNewParam 구조 |
|------|---------|-----------------|
| 불법TM수신차단 | - | 스팸차단번호 파라미터 |
| 번호도용차단 | - | KISA 동의여부(Y/N) |

팝업은 프론트엔드 `McpAdditionEditPop.vue`에서 처리하며, `additionReg` 호출 시 `ftrNewParam`에 팝업 입력값을 담아 전송. 백엔드는 그대로 X21에 전달.

## 4. 처리 흐름 상세

### 4.1 가입자정보조회 흐름 (join-info)

```
[프론트 — ChangeTypeCust.vue]
  고객 이름 + 휴대폰번호 입력 후 인증 버튼
  → POST /api/v1/join-info { name, ctn }

[백엔드 — SvcChgRestController.joinInfo()]
  1. ContractInfoMapper.selectContractInfo(name, ctn)
     → MSP_JUO_SUB_INFO@DL_MSP(M전산)
     → ncn(계약번호 9자리), custId 반환
  2. MplatFormSvc.contractValdChk(clientIp, cIntUsrId, ncn, ctn, custId) → Y04
     → 유효 여부(true/false) 반환 (고객유형·birthDt 등은 X01에서 조회)
  3. MplatFormSvc.perMyktfInfo(ncn, ctn, custId) → X01
     → 이메일, 주소, 가입일 등 추가정보
  4. JoinInfoResVO 구성 후 반환

[프론트 — JoinInfoResVO 수신]
  → Pinia store(service_change_form)에 ncn, ctn, custId, custTypeCd 저장
  → 다음 단계 활성화
```

## 5. DB 연동 (M전산)

### 5.1 ContractInfoMapper — selectContractInfo

```sql
-- ASIS: MypageMapper.xml - selectContractNum 동일 구조
-- ⚠ 로컬 개발: PostgreSQL → LIMIT 1 사용
-- ⚠ 운영 환경: Oracle(M전산) → ROWNUM = 1 사용 (Oracle linked server 경유)
SELECT CONTRACT_NUM as ncn, CUST_ID as custId
FROM MSP_JUO_SUB_INFO
WHERE SUBSCRIBER_NO = #{ctn}
  AND SUB_LINK_NAME = #{name}
LIMIT 1
-- Oracle 운영: LIMIT 1 → AND ROWNUM = 1 로 교체
```

> 로컬 DB는 PostgreSQL(`application.properties` 확인), 운영은 Oracle DB Link 경유. MyBatis databaseId 또는 환경별 XML 분리 검토 필요.

### 5.2 SvcChgBaseMapper — 서비스변경 업무 코드

```xml
<!-- MSF_CD_GROUP_BAS: 서비스변경 업무 그룹 (SVC_TGT_CD 기준) -->
SELECT GROUP_CD, GROUP_NM, SVC_TGT_CD, SORT_SEQ, USE_YN
FROM MSF_CD_GROUP_BAS
WHERE USE_YN = 'Y'
ORDER BY SORT_SEQ

<!-- MSF_CD_DTL: 상세 코드 -->
SELECT DTL_CD, DTL_CD_NM, CONCURRENT_BLOCK_YN, IMAGING_YN, SORT_SEQ
FROM MSF_CD_DTL
WHERE GROUP_CD = #{groupCd} AND USE_YN = 'Y'
ORDER BY SORT_SEQ
```

> `MSF_CD_GROUP_BAS`, `MSF_CD_DTL` 테이블이 스마트서식지 DB에 없으면 DDL 추가 필요. 테이블 정의서(`스마트서식지-DS-05-테이블 정의서`) 확인 후 생성.
