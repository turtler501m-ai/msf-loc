## 1. 문서 목적과 사용법

H4는 “명확한 불필요 기능”을 빠르게 식별하기 위한 문서다.  
H5는 그 다음 단계 문서로, 실제 정리 작업 시 아래 판단을 돕기 위해 작성한다.

1. 지금 바로 삭제해도 되는가
2. 당장은 남기되 전환 대상에서는 제외해야 하는가
3. 기능 자체는 필요하지만 ASIS 복사 구현을 재사용하면 안 되는가
4. H2의 일반 규칙과 달리 H1에서 실제 사용 대상으로 살아 있는 예외는 무엇인가

본 문서는 특히 H2의 “화면 진입 함수 제외” 원칙을 그대로 적용하지 않고, H1에서 실제 TOBE 대상으로 유지된 항목은 예외로 분리한다.

---

## 2. 최종 분류 기준

### 2-1. A등급: 즉시 삭제 가능

아래 조건을 만족하면 삭제 우선순위가 가장 높다.

- H1에서 `🚫 제외`로 명시됨
- H2에서 `ASIS 전용`, `TOBE 범위 외`, `팝업`, `탭목록`, `배치`로 분류됨
- 현재 TOBE API/화면/업무흐름 어디에도 직접 연결되지 않음

### 2-2. B등급: 전환 제외 보관 가능

아래 조건이면 코드 삭제보다 별도 격리/보관이 적절하다.

- 현재 TOBE 핵심기능과는 무관하지만 운영 참고 가능성이 있음
- 배치/관리성 기능이라 추후 별도 설계 여부가 남아 있음

### 2-3. C등급: ASIS 복사 재사용 금지

기능 자체는 필요하지만, ASIS 구현을 그대로 복사해 쓰면 안 되는 경우다.

- 외부 연동 구조가 변경됨
- TOBE 테이블 구조가 변경됨
- 포시에스/신규 인증/공통 API 구조로 다시 구현해야 함

### 2-4. D등급: 삭제 금지 예외

아래 조건이면 화면 메서드라도 본 문서의 삭제 대상이 아니다.

- H1에서 `✅ 완료` 또는 `⚠️ 확인필요`로 유지되고 있음
- E1 기준 TOBE 기능번호가 이미 매핑되어 있음
- 현재 레거시 URL 호환 또는 화면 진입 유지가 필요함

## 6. 실제 코드 정리 순서

1. H1에서 `🚫 제외`로 확정된 메서드만 1차 후보로 추린다.
2. H4/H5의 `A등급`과 대조해 즉시 삭제 목록을 확정한다.
3. 삭제 대상 메서드가 URL 매핑, JSP, 프론트 호출, 배치 설정에 남아 있는지 확인한다.
4. KT 결합 계열처럼 패키지 단위 삭제가 가능한 영역은 개별 메서드보다 패키지 단위로 정리한다.
5. Controller뿐 아니라 연관 Service/Mapper까지 같이 추적해 미사용 메서드를 함께 정리한다.
6. `B등급`은 삭제하지 말고 `legacy`, `backup`, `not-used` 성격으로 분리 보관할지 결정한다.
7. `C등급`은 삭제 대상이 아니라 “ASIS 복사 재사용 금지” 대상으로 관리한다.

## 7. 실제 확인 체크리스트

### 7-1. 삭제 전 체크

- 해당 메서드가 H1에서 `✅ 완료` 또는 `⚠️ 확인필요`로 매핑되지 않았는가
- 동일 이름 또는 유사 이름의 Service 메서드가 다른 현재 API에서 재사용되지 않는가
- 프론트 호출 URL이 남아 있지 않은가
- Spring `@RequestMapping`만 남고 실제 호출이 사라진 상태인가
- 배치 또는 운영 스크립트에서 참조하지 않는가
- 동일 기능이 공통 API로 이미 대체되었는가

### 7-2. 삭제 후 체크

- 중복 URL 매핑이 사라졌는가
- 미사용 Service/Mapper/DTO도 함께 정리되었는가
- 더 이상 사용하지 않는 DTO/VO/Mapper도 같이 정리할 수 있는가
- 문서 H1/H4/H5와 코드 상태가 불일치하지 않는가
- 범위 외 기능 제거로 인해 남은 컨트롤러의 책임이 더 명확해졌는가

## 8. 문서별 역할 정리

| 문서 | 역할 |
|------|------|
| `H1` | E1 기준 TOBE 구현/매핑 기준서 |
| `H2` | 실 ASIS 소스 기준 함수 인벤토리와 필요/불필요 1차 분류 |
| `H4` | 명확한 불필요 기능 요약본 |
| `H5` | 실제 코드 정리용 상세 가이드와 예외/우선순위 문서 |

---

## 9. 권고사항

1. 실제 소스 정리는 반드시 H1과 같이 봐야 하며, H2의 “화면 진입 제외” 일반 규칙만으로 삭제 판단하면 안 된다.
2. 삭제 작업은 메서드 단위보다 `팝업`, `KT 결합`, `범위 외 고객요청`처럼 기능군 단위로 진행하는 편이 안전하다.
3. 보험/해지/명의변경처럼 화면 진입 메서드가 아직 살아 있는 영역은 프론트 완전 이관 전까지 삭제 대상에서 제외한다.
4. SCAN/XML 원형 로직은 유지 여부 문제가 아니라 재구현 문제이므로 삭제 문맥과 재구현 문맥을 분리해서 관리해야 한다.
